    网络层：数据平面¶

    一、网络层概述¶

    网络层（Network Layer）是 OSI/TCP-IP 体系结构中负责端到端数据传输的层次，位于传输层（Transport Layer）之下、数据链路层（Data Link Layer）之上。

    核心职责¶

    • 发送端：将传输层的报文段（Segment）封装成数据报（Datagram），并向下传递给数据链路层。
    • 接收端：将数据报解封装，将报文段向上交付给传输层。
    • 网络层协议存在于每一台主机和每一台路由器中。
    • 路由器检查并处理所有经过其的 IP 数据报的首部字段。

    位置与协议栈关系¶

    1
2
3
4
5
    应用层 (Application)
传输层 (Transport)
网络层 (Network)      <- 本节重点
数据链路层 (Data Link)
物理层 (Physical)

    二、网络层的两大核心功能¶

    1. 转发（Forwarding）— 数据平面¶

    • 定义：在单个路由器内，将到达输入端口的数据报，根据转发表（Forwarding Table）转发至相应的输出端口。
    • 范围：局部动作（local, per-router function）。
    • 实现：通常由硬件完成（ASIC / 网络处理器），以满足线速（Line-rate）处理需求。
    • 类比：进入苏州站、从苏州站离开——即在单个节点内的通过动作。
    Warning

    交换机是二层设备，主要依据 MAC 地址转发帧；

    路由器是三层设备，主要依据 IP 地址转发分组；

    路由器不是数据链路层设备。

    在今天的局域网环境中，作为链路层互联设备，实际部署里基本都是交换机，现代的交换机已经具备三层的功能。

    2. 路由（Routing）— 控制平面¶

    • 定义：在网络范围内，通过路由算法计算从源主机到目的主机的端到端转发路径，并生成转发表。
    • 范围：全局逻辑（network-wide logic），涉及多个节点的协同。
    • 实现：传统方式由路由器软件实现；SDN 方式由远程控制器实现。
    • 类比：规划从南京到上海的路线（南京—无锡—苏州—上海）。

    三、数据平面与控制平面¶

    维度 数据平面（Data Plane） 控制平面（Control Plane）
    功能 数据报的转发 路由表/转发表的计算
    范围 单路由器本地 全网范围
    时延要求 极低（纳秒级） 相对宽松（秒级可接受）
    实现方式 硬件（ASIC） 软件（路由协议/SDN 控制器）

    控制平面的两种实现方式¶

    传统路由算法（Per-Router Control Plane）¶

    每台路由器内嵌独立的路由算法模块（如 OSPF、BGP），各路由器之间通过路由协议报文相互通信，协同维护路由信息。路由计算结果写入本地转发表，数据平面据此转发。

    软件定义网络（Software-Defined Networking, SDN）¶

    控制逻辑从路由器中剥离，集中到远程的SDN 控制器（Controller）。控制器统一计算全网转发策略，通过南向接口（如 OpenFlow）将转发表下发（Install）到各路由器。路由器本身退化为纯转发设备（Data Plane only），内置控制代理（Control Agent, CA）。

    优势：集中视图、便于编程管理、快速部署新协议。

    缺点：控制器可能成为单点故障和性能瓶颈，控制平面与数据平面分离带来额外时延和通信开销，系统实现与运维更复杂，同时还存在安全、兼容性以及流表容量受限等问题。

    四、网络服务模型¶

    问题背景¶

    网络服务模型（Network Service Model）描述了网络层为"信道"（从发送方到接收方传输数据报）所提供的服务质量保证，即从主机视角看到的服务质量（Quality of Service, QoS）。

    单包服务示例¶

    服务类型 含义
    保证交付（Guaranteed Delivery） 数据报一定能到达目的地
    有界时延交付（Guaranteed Delivery with bounded delay） 在指定时延（如 40ms）内交付

    流（Flow）级别服务示例¶

    服务类型 含义
    保序交付（In-order delivery） 数据报按发送顺序到达
    最小带宽保证（Guaranteed minimum bandwidth） 为某条流保证带宽下限
    分组间隔约束（Inter-packet spacing restrictions） 控制报文到达间隔，降低抖动（Jitter）

    ATM 网络服务模型（对比）¶

    ATM（Asynchronous Transfer Mode）提供多种服务类别，按优先级递减排列：

    服务类别 带宽保证 无丢失保证 有序保证 时序保证 拥塞反馈
    CBR（恒定比特率） 恒定 是 是 是 不拥塞
    VBR（可变比特率） 保证 是 是 是 不拥塞
    ABR（可用比特率） 保证最小 否 是 否 有反馈
    UBR（未指定比特率） 无 否 是 否 无

    Internet（IP）的服务模型：尽力而为（Best Effort）¶

    IP 网络不提供任何保证：

    属性 Internet（IP）
    带宽保证 无
    丢包保证 无
    有序交付 不保证（任意顺序可能）
    时延保证 无
    拥塞反馈 无（通过丢包推断）

    尽力而为服务成功的原因¶

    1. 机制简单：降低了路由器实现难度，推动了 Internet 的大规模部署。
    2. 充足带宽供给：对于实时应用（语音、视频），带宽充裕时，"大多数时候"性能足够好。
    3. 应用层分布式服务：CDN、数据中心靠近用户，从多个位置提供服务，弥补了网络层的不足。
    4. 弹性服务的拥塞控制：TCP 拥塞控制机制有效防止网络崩溃。

    五、路由器¶

    基本概念¶

    路由器是 Internet 基础设施的核心构建块（Core Building Block）。路由器市场规模超过 1200 亿美元，主要厂商（思科、华为、瞻博、阿尔卡特-朗讯）占据 90% 以上市场份额。

    路由器容量定义¶

    \[\text{Router Capacity} = N \times R\]
    • \(N\)：外部端口数量（Number of external ports）
    • \(R\)：单端口线速（Line rate per port）
    Warning

    记得要乘\(N\)！

    路由器类型与规格¶

    类型 单端口速率 \(R\) 总容量 \(NR\) 典型场景
    核心路由器（Core Router） 10/40/100/200/400 Gbps O(100) Tbps（汇聚） 运营商骨干网
    边缘路由器（Edge Router） 1/10/40/100 Gbps O(100) Gbps ISP 接入层
    小型商用路由器（Small Business） 1 Gbps < 10 Gbps 家庭/小企业

    六、路由器内部架构¶

    路由器在逻辑上分为两个平面：

    两大核心功能：

    1. 运行路由算法/协议（控制平面）
    2. 将数据报从输入端口转发至输出端口（数据平面）

    七、输入端口（Input Port）¶

    处理流水线¶

    1
    物理层（比特接收）-> 数据链路层（协议解封装/差错检测）-> 查找转发 & 排队 -> 交换矩阵

    主要任务¶

    1. 接收数据报：物理层比特接收、数据链路层解封装。
    2. 更新 IP 首部：TTL 递减、重新计算 Checksum、处理 Options 和分片（Fragmentation）。
    3. 目的地址查找（Destination Address Lookup）：根据目的 IP 地址，在本地转发表中查找输出端口。
    4. 排队（Queuing）：当数据报到达速率超过交换矩阵接受速率时进行缓存。

    主要挑战：速度¶

    • 在 40 Gbps 线速下，最小 100 字节的数据报每隔 20 纳秒到达一次。
    • 必须使用专用 ASIC（Application-Specific Integrated Circuit） 或网络处理器实现。

    最长前缀匹配（Longest Prefix Matching, LPM）¶

    核心思想：转发表中不为每个 IP 地址存储一条记录（否则 IPv4 需 40 亿条），而是以地址前缀（Prefix）为单位聚合。查找时，选择与目的地址匹配的最长前缀对应的输出端口。

    示例（8 位地址简化）：

    目的地址前缀 输出端口
    110000** Port 1
    110001** Port 2
    11001*** Port 3
    1101**** Port 4
    • 数据报目的地址 11000001（Port 2）：匹配 110001**（6位）比 11*（2位）更长，选 Port 2。

    高效查找：Trie（前缀树）

    • 线性扫描所有表项平均复杂度为 \(O(\text{条目数})\)，不可接受。
    • 利用二进制字符串的树形结构，构造二叉 Trie 树。
    • 从根出发，按目的地址逐比特走树，记录沿途最后一次匹配到的端口，到达叶节点（或无路可走）时输出该端口。
    • 复杂度：\(O(\text{地址位数})\) = \(O(32)\)（IPv4）或 \(O(128)\)（IPv6），与条目数无关。
    • 硬件实现：TCAM（Ternary Content Addressable Memory）可实现单周期并行匹配。

    八、交换矩阵（Switching Fabric）¶

    交换矩阵负责将数据报从输入缓存传输到对应的输出缓存。

    交换速率（Switching Rate）：通常要求达到输入/输出线速的 \(N\) 倍（\(N\) 为端口数），以避免输入端口排队。

    三种典型实现¶

    1. 基于内存的交换（Switching via Memory）¶

    • 第一代路由器实现方式，类似传统计算机架构。
    • 数据报从输入端口经系统总线复制到 CPU 内存，再由 CPU 控制写入输出端口。
    • 瓶颈：每个数据报需跨越总线 2 次（读入 + 写出），速度受限于内存带宽。
    • 带宽上限约为内存总线带宽的一半。

    2. 基于总线的交换（Switching via Bus）¶

    • 输入端口通过共享总线将数据报直接传输至输出端口内存（无需 CPU 介入）。
    • 总线争用（Bus Contention）：同一时刻只能有一个数据报占用总线。
    • 瓶颈：总线带宽即为交换上限。
    • 典型：Cisco 5600，总线带宽 32 Gbps，适合接入/企业级路由器。

    3. 基于互联网络的交换（Switching via Crossbar/Interconnection Network）¶

    • 采用纵横开关（Crossbar Switch）或 Banyan 网络等多级互联网络。
    • 每对（输入端口, 输出端口）之间可同时传输，支持并行交换，突破总线带宽限制。
    • 高级设计：将数据报切分为固定长度的信元（Cell），通过交换矩阵进行信元级交换。
    • 典型：Cisco 12000，互联网络带宽 60 Gbps。
    交换方式 并行度 瓶颈 适用规模
    内存 无 内存带宽 早期/低速
    总线 无 总线带宽 中小型
    交叉矩阵 高 端口数/调度 高性能核心路由器

    九、输出端口（Output Port）¶

    处理流水线（反向）¶

    1
    交换矩阵 -> 分类（Classifier）-> 缓冲管理（Buffer Management）-> 调度（Scheduler）-> 数据链路层封装 -> 物理层发送

    三大功能模块¶

    模块 功能
    包分类（Packet Classification） 将数据报映射到不同流（Flow）/优先级队列
    缓冲管理（Buffer Management） 决定何时、丢弃哪个数据报
    调度（Scheduler） 决定何时、从哪个队列发送数据报

    分类字段¶

    IP 数据报可根据以下首部字段进行分类（多维范围查找）：

    • 源/目的 IP 地址（32 bits）
    • 源/目的 TCP/UDP 端口号（16 bits）
    • 服务类型 TOS（8 bits）
    • 协议类型（8 bits）

    十、排队与调度¶

    输入端口排队与 HOL 阻塞¶

    产生原因：当交换矩阵的速率低于所有输入端口的总到达速率时，输入端口产生队列积压。

    队头阻塞（Head-of-Line Blocking, HOL Blocking）：

    • 队列头部的数据报（因目的端口冲突）无法被调度进入交换矩阵，同时阻塞了队列后面本可被发送的数据报。
    • HOL Blocking 会显著降低输出端口的实际吞吐量（理论上限约为线速的 58%）。
    Info

    队头阻塞在出现“反压”的时候是最严重的。

    输出端口排队¶

    产生原因：当交换矩阵的输出速率（NR）超过链路发送速率（R）时，输出端口产生积压。

    后果：数据报可能因缓冲区溢出而丢失（Packet Loss），产生排队时延（Queuing Delay）。

    缓冲区大小¶

    RFC 3439 经验规则：

    \[B = \text{RTT} \times C\]

    其中 \(C\) 为链路容量，典型 RTT = 250 ms。例如，10 Gbps 链路需要 2.5 Gbit 缓冲。

    更新建议（考虑多流场景）：

    \[B = \frac{\text{RTT} \times C}{\sqrt{N}}\]

    \(N\) 为并发流数量。缓冲过大会导致缓冲膨胀（Bufferbloat），增加时延，对实时应用和 TCP 拥塞控制均有害。

    Warning

    1. 缓冲区太小会导致丢包

      当短时间内到达的数据超过链路当前能发出去的数据时，数据包就需要先进入缓冲区等待。如果缓冲区太小，很快就会被占满，后续到达的包只能被直接丢弃。这样会导致网络丢包增加，TCP 会误以为发生了严重拥塞，从而降低发送速率，最后使链路利用率下降。所以缓冲区不能太小，因为它必须具备吸收突发流量的能力。

    2. 缓冲区太大会导致时延增大

      缓冲区过大时，虽然包不容易被丢弃，但这些包会在队列中等待很长时间才能被发送出去，结果就是排队时延显著增加，RTT 变大。这样会严重影响语音、视频、在线游戏等对时延敏感的实时应用，同时也会让 TCP 对网络拥塞的感知和反馈变慢，使整个网络表现变得迟钝。所以缓冲区不能太大，因为过大的缓冲区会把问题从“丢包”变成“高延迟”。

    3. 缓冲区大小适中的本质是平衡吞吐率和时延

      设置缓冲区的目标，不是单纯追求更少丢包，也不是单纯追求更低延迟，而是在两者之间取得平衡。缓冲区需要大到足以吸收正常的流量波动，避免链路空闲和频繁丢包；同时又要小到不会让过多数据长期堆积在队列中，避免时延过高。也就是说，理想状态是让瓶颈链路始终保持忙碌，但队列不要过长，这就是为什么缓冲区大小必须适中。

    缓冲管理策略（Buffer Management）¶

    丢包策略（Drop Policy）¶

    策略 描述
    尾部丢弃（Tail Drop） 缓冲满时直接丢弃新到达的数据报（最简单）
    优先级丢弃（Priority Drop） 按优先级选择丢弃低优先级数据报
    随机早期检测（RED, Random Early Detection） 缓冲区达到阈值时开始随机丢包，提前向发送方发送拥塞信号，避免全局同步

    标记策略（Marking Policy）¶

    • ECN（Explicit Congestion Notification）：在数据报首部标记拥塞信号，通知接收方转告发送方降速，比丢包更温和。

    调度算法（Scheduling Algorithms）¶

    1. FIFO（First-In, First-Out）¶

    • 最简单，按到达顺序发送，无分类。
    • 配合尾部丢弃缓冲管理构成最基础的路由器。
    • 缺点：无优先级、无公平性保证。

    2. 优先级调度（Priority Queuing）¶

    • 将数据报分类到不同优先级队列。
    • 始终优先服务高优先级队列（非抢占式或抢占式）。
    • 同一优先级队列内采用 FCFS（First-Come, First-Served）。
    • 缺点：低优先级流可能出现饥饿（Starvation）。

    3. 轮询调度（Round Robin, RR）¶

    • 将数据报分类到多个队列，调度器循环轮询每个队列，每次发送一个完整数据报（若该队列非空）。
    • 各队列获得近似等带宽共享。
    • 缺点：不能区分重要性差异，不保证带宽比例。

    4. 加权公平排队（Weighted Fair Queuing, WFQ）¶

    • 轮询调度的泛化版本，为每个队列 \(i\) 分配权重 \(w_i\)。
    • 每轮中队列 \(i\) 获得的服务比例为：
    \[\frac{w_i}{\sum_j w_j}\]
    • 为每类流量提供最小带宽保证（per-class minimum bandwidth guarantee）。
    • 是实现 QoS 差异化服务（DiffServ）的核心机制。

    各字段详解¶

    字段 位数 说明
    Version 4 IP 版本号，IPv4 = 4，IPv6 = 6
    IHL（Internet Header Length） 4 首部长度，单位为 32 位字（4 字节），最小值 5（20 字节），最大值 15（60 字节）
    Type of Service (TOS) 8 服务类型：Precedence（3 bit，8 个优先级）+ Reliability（1 bit）+ Delay（1 bit）+ Throughput（1 bit）+ 保留位
    Total Length 16 整个数据报（首部+数据）的字节数，最大 65,535 字节
    Identification 16 序列号，与源地址、目的地址、用户协议共同唯一标识一个数据报（用于分片重组）
    Flags 3 第 0 位保留；DF（Don't Fragment）：禁止分片；MF（More Fragments）：更多分片
    Fragment Offset 13 分片偏移量，单位为 8 字节，标识该分片在原数据报数据部分的位置
    Time to Live (TTL) 8 生存时间，每经过一台路由器递减 1，降至 0 时路由器丢弃该数据报并返回 ICMP 超时报文
    Protocol 8 指示上层协议：TCP = 6，UDP = 17，ICMP = 1，OSPF = 89
    Header Checksum 16 仅对首部做反码和（1's complement sum）校验；每个路由器修改 TTL 后需重新计算，计算时将本字段置 0
    Source Address 32 源 IP 地址
    Destination Address 32 目的 IP 地址
    Options 可变（≤ 40 字节） 可选字段，包括记录路由、时间戳、严格/松散源路由等

    数据字段（Data Field）¶

    • 携带来自上层（传输层）的有效载荷，如 TCP 报文段或 UDP 数据报。
    • 长度必须是 8 比特（1 字节）的整数倍（Octet 对齐）。
    • 数据报（首部 + 数据）最大长度：65,535 字节。

    为什么首部 Checksum 每跳都要重新计算？¶

    因为路由器每次转发数据报时都会将 TTL 字段减 1，导致首部内容发生变化，原有 Checksum 失效，必须重新计算后再转发，以保证下一跳可以正确检验首部完整性。

    总结¶

    主题 核心要点
    网络层功能 转发（数据平面）+ 路由（控制平面）
    控制平面 传统分布式（Per-Router）或 SDN 集中式
    服务模型 Internet = 尽力而为；ATM = CBR/VBR/ABR/UBR
    路由器架构 输入端口 → 交换矩阵 → 输出端口
    地址查找 最长前缀匹配（LPM）+ Trie 树/TCAM
    交换方式 内存 / 总线 / 交叉矩阵（性能递增）
    调度算法 FIFO / Priority / RR / WFQ（灵活性与复杂度递增）
    IPv4 首部 20 字节固定 + 可变 Options，关键字段：TTL、Protocol、Checksum、Src/Dst
