    同步：进阶¶

    一、互斥锁、条件变量与信号量的关系回顾¶

    1.1 三者关系总结¶

    工具 能力 限制
    互斥锁（Mutex） 保护临界区，实现互斥 只能表达"进/出"，无法表达复杂条件
    信号量（Semaphore） 可视为高级互斥锁；支持线程控制（计数） 难以优雅实现条件变量；泛化易出错（Lampson警告）
    条件变量（Condition Variable） 适用于任意同步条件；与互斥锁配合可实现信号量 必须配合互斥锁使用

    Lampson 警告：小心过度泛化同步原语——信号量虽强大，但在某些场景（如实现条件变量）下容易写出错误的代码。

    二、读者-写者问题（Readers–Writers Problem）¶

    2.1 问题描述¶

    • 多个线程需要读取共享数据，一个或多个线程需要写入共享数据。
    • 目标：在保证正确性的前提下，允许尽可能多的并发。

    2.2 同步规则¶

    操作组合 是否需要互斥 原因
    读 - 读 ❌ 不需要 读操作不改变数据，无竞争
    写 - 写 ✅ 需要 写操作互相干扰
    写 - 读 ✅ 需要 写者写入时，读者不能读脏数据

    简单对所有线程加互斥锁虽然正确，但会限制读者之间不必要的串行化，性能差。

    三、读写锁的三种实现¶

    3.1 实现一：读者优先（Reader Preference）¶

    核心思路：用信号量 lock 保护读者计数，用 rwlk 作为进入临界区的锁。第一个读者获取 rwlk，最后一个读者释放 rwlk。写者直接竞争 rwlk。

    typedef struct _rwlock_t {
    sem_t lock;    // 保护自身锁状态（读者计数）
    sem_t rwlk;    // 进入临界区的锁
    int readers;
} rwlock_t;

void acquire_readlock(rwlock_t *rw) {
    P(&rw->lock);
    if (rw->readers == 0)
        P(&rw->rwlk);   // 第一个读者抢占 rwlk
    rw->readers++;
    V(&rw->lock);
}

void release_readlock(rwlock_t *rw) {
    P(&rw->lock);
    rw->readers--;
    if (rw->readers == 0)
        V(&rw->rwlk);   // 最后一个读者释放 rwlk
    V(&rw->lock);
}

void acquire_writelock(rwlock_t *rw) { P(&rw->rwlk); }
void release_writelock(rwlock_t *rw) { V(&rw->rwlk); }

    问题：若持续有读者进入，rwlk 永远不会被释放，写者饿死（Starvation）。

    3.2 实现二：写者优先（Writer Preference）¶

    核心思路：引入 tryRead 信号量。第一个写者到来时，占用 tryRead，阻止后续读者进入；最后一个写者离开时释放 tryRead。

    typedef struct _rwlock_t {
    sem_t rlock;    // 读操作保护锁
    sem_t wlock;    // 写操作保护锁
    sem_t rwlk;     // 进入临界区的锁
    sem_t tryRead;  // 读者进入前先看是否有等待写者
    int readers;
    int writers;    // 等待/活跃的写者数量
} rwlock_t;

void acquire_readlock(rwlock_t *rw) {
    P(&rw->tryRead);  // 若有写者，在此阻塞
    P(&rw->rlock);
    if (rw->readers == 0) P(&rw->rwlk);
    rw->readers++;
    V(&rw->rlock);
    V(&rw->tryRead);  // 释放后续读者可继续排队
}

void acquire_writelock(rwlock_t *rw) {
    P(&rw->wlock);
    if (rw->writers == 0)
        P(&rw->tryRead);  // 第一个写者阻断读者
    rw->writers++;
    V(&rw->wlock);
    P(&rw->rwlk);
}

void release_writelock(rwlock_t *rw) {
    V(&rw->rwlk);
    P(&rw->wlock);
    rw->writers--;
    if (rw->writers == 0)
        V(&rw->tryRead);  // 最后一个写者放开读者
    V(&rw->wlock);
}

    问题：若持续有写者，读者饿死。

    3.3 实现三：公平锁（Ticket Lock 排队）¶

    核心思路：引入 ticket lock 维护先进先出队列，线程按到达顺序获得进入机会，避免单方饿死。

    typedef struct lock_ticket {
    int ticket;  // 当前发放的最大票号
    int turn;    // 当前轮到的票号
} ticket_t;

void ticket_lock(ticket_t *tlk) {
    int myturn = __sync_fetch_and_add(&tlk->ticket, 1);
    while (tlk->turn != myturn)
        ;  // 自旋等待
}

void ticket_unlock(ticket_t *tlk) {
    __sync_fetch_and_add(&tlk->turn, 1);
}

    typedef struct _rwlock_t {
    sem_t rlock;
    sem_t rwlk;
    ticket_t queue;  // FIFO 队列
    int readers;
} rwlock_t;

void acquire_readlock(rwlock_t *rw) {
    ticket_lock(&rw->queue);
    P(&rw->rlock);
    if (rw->readers == 0) P(&rw->rwlk);
    rw->readers++;
    V(&rw->rlock);
    ticket_unlock(&rw->queue);
}

void acquire_writelock(rwlock_t *rw) {
    ticket_lock(&rw->queue);
    P(&rw->rwlk);
    ticket_unlock(&rw->queue);
}

    效果：线程按到达顺序排队，读者和写者公平竞争，不会出现单方饿死。

    四、Read-Copy-Update（RCU）¶

    4.1 背景：锁的性能瓶颈¶

    • 互斥锁上锁/释放需要的 CPU cycles 比普通指令多一个数量级。
    • 在短临界区场景下，锁本身成为并发程序的主要瓶颈——线程越多，反而性能越低。
    • 读写锁解决了"读-读"并发，但"读-写"之间仍需同步开销。

    4.2 一致性模型¶

    模型 含义
    强一致性（Strong Consistency） 数据更新后，其他线程立即可见
    最终一致性（Eventual Consistency） 数据更新后，其他线程经过一段时间后最终可见

    RCU 采用最终一致性：读者可能短暂读到旧数据，但系统最终一致，且绝对不会读到中间状态（写的原子性保证）。

    4.3 RCU 核心特性¶

    • 读者：可与写者并发运行，不被阻塞，读操作几乎零开销。
    • 写者：需要额外成本（两阶段更新）。
    • 适用场景：read-mostly（读多写少），如路由表、用户组信息等操作系统内核对象。

    4.4 两阶段更新：Removal 与 Reclamation¶

    阶段 操作 并发性
    Removal（移除） 删除数据结构中旧项的引用，将引用指向新项 可与读者完全并发
    Reclamation（回收） 真正释放/清除旧数据的内存 需要同步：等所有旧读者结束后才能执行

    RCU 一大特性：需要维护数据的多个版本，读者可以读到旧版本数据。

    链表更新示例（更新节点 d → d'）：

  4. 释放旧节点 d 的内存

    4.5 订阅-发布机制（Publish-Subscribe）：保证 Removal 原子性¶

    问题：在编译器优化和弱内存模型下，指令重排可能导致读者读到不完整的新数据（中间状态）。

    解决方案：内存屏障（Memory Barrier），Linux RCU 将其封装为 API：

// 读者：订阅数据
foo *p = rcu_dereference(gp);    // 包含读内存屏障，保证读到的指针有效

    void foo_update() {
    mutex_lock(&lk);
    new_gp = kmalloc(sizeof(*p), GFP_KERNEL);
    new_gp->a = 1;

    new_gp->c = 3;

    rcu_read_lock();
    foo *p = rcu_dereference(gp);
    if (p != NULL)
        do_something(p->a, p->b, p->c);
    rcu_read_unlock();
}

    4.6 宽限期（Grace Period）¶

    定义：从写者发布更新，到所有"旧"读者（在更新前就开始读的线程）都结束读取的这段时间。

    规则： - 读者用 rcu_read_lock() / rcu_read_unlock() 显式标记读的区间。 - 写者用 synchronize_rcu() 阻塞等待宽限期结束，之后才能释放旧数据。 - 在宽限期结束后开始的读者（新读者）一定读到新数据，不影响回收。

    时序示意：

    1
2
3
4
5
6
7
    T0: 写者发布更新
│
├── R1（旧读者）：rcu_read_lock ... rcu_read_unlock  ─────────┐
├── R2（旧读者）：rcu_read_lock ........ rcu_read_unlock ─────┤ 宽限期
├── R3（新读者）：          rcu_read_lock .. rcu_read_unlock  │
│                                                           │
└────────────────────────────── 宽限期结   可安全释放旧数据─── ←┘

    经典 RCU 宽限期实现（内核）：

    • rcu_read_lock()：关中断（禁止抢占）。
    • rcu_read_unlock()：开中断（恢复抢占）。
    • 写者检测到所有 CPU 都经历了一次 context switch，即可确认所有旧读者已结束。

    异步回收：若写者不想阻塞，可用 call_rcu(callback) 注册回调函数，宽限期结束后自动调用，无需等待。

    用户态 RCU：也存在用户态实现（User-space RCU，liburcu），使用信号或其他机制模拟宽限期检测。

    4.7 RCU 在 Linux 内核中的地位¶

    • Linux 2.5（2002年）引入。
    • 2024年 Linux 内核中 10% 以上的锁基于 RCU。
    • 比传统读写锁具有更高的可扩展性（highly scalable）。

    五、哲学家就餐问题（Dining Philosophers Problem）¶

    5.1 问题描述¶

    • 由 Dijkstra（1965）提出，Hoare 修改为如今的版本。
    • 5 位哲学家围坐圆桌，每人轮流思考和吃饭。
    • 每两位哲学家之间放一根筷子（共 5 根），哲学家必须同时持有左右两根筷子才能吃饭。
    • 目标：无死锁、无饿死、并发度高。

    5.2 辅助定义¶

    #define N 5
sem_t forks[N];
int left(int i)  { return i; }
int right(int i) { return (i + 1) % N; }

    void philosopher(int i) {

    5.5 方案三：全局限流（信号量容量控制）¶

    思路：5 根筷子最多允许 4 个人同时尝试，根据鸽巢原理，至少有一人能拿到两根筷子。

    sem_t capacity;  // 初始化为 N-1 = 4

void philosopher(int i) {
    while (1) {
        think();
        P(capacity);
        P(forks[left(i)]);
        P(forks[right(i)]);
        eat();
        V(forks[left(i)]);
        V(forks[right(i)]);
        V(capacity);
    }
}

    正确性：至多 4 人同时持有一根筷子，5 根筷子中至少有一人得到两根 → 不会死锁。

    并发度：最多 ⌊N/2⌋ = 2 人同时吃饭（N=5 时）。

    六、死锁与饿死¶

    6.1 基本概念¶

    概念 定义
    安全性（Safety） 系统不会进入错误状态（互斥性、数据一致性）
    活性（Liveness） 系统最终能推进（每个线程最终都能完成）
    饿死（Starvation） 某线程在有限时间内无法继续推进
    死锁（Deadlock） 多个线程形成循环等待，每个线程都等待环中其他线程释放资源

    注意：死锁和饿死都破坏活性，但不破坏安全性（临界区没有被多个线程同时进入）。

    6.2 死锁与饿死的关系¶

    1
    死锁 ⊆ 饿死
    • 死锁一定导致饿死（等待条件永远无法满足）。
    • 饿死不一定是死锁（可能只是调度不公平，运气差一直被抢占）。

    6.3 死锁的四个必要条件（Coffman 条件，PPT 未列出，补充）¶

    发生死锁必须同时满足以下四个条件：

    条件 含义
    互斥（Mutual Exclusion） 资源一次只能被一个线程持有
    持有并等待（Hold and Wait） 线程持有资源的同时等待其他资源
    不可抢占（No Preemption） 资源只能由持有者主动释放
    循环等待（Circular Wait） 线程间形成环形等待链

    破解策略（PPT 未列出，补充）：

    • 破坏互斥：使用无锁算法（如 RCU）。
    • 破坏持有并等待：一次性申请所有资源（方案三：全局限流）。
    • 破坏不可抢占：使用 trylock（方案二）。
    • 破坏循环等待：资源编号排序（方案四）。

    七、补充：非阻塞算法（Non-blocking Algorithm）¶

    7.1 分类（PPT 未列出）¶

    级别 含义
    Wait-free 每个线程在有限步内完成操作，不受其他线程影响
    Lock-free 整个系统保证有线程能推进，但个别线程可能重试
    Obstruction-free 单独运行时能完成操作

    RCU 的读者端是 wait-free 的（几乎零开销，不等待任何人）。

    7.2 常见无锁原语¶

    • CAS（Compare-and-Swap）：__sync_bool_compare_and_swap(ptr, old, new)，是实现无锁数据结构的基础。
    • FAA（Fetch-and-Add）：__sync_fetch_and_add(ptr, val)，Ticket Lock 使用的原语。
    • 内存屏障（Memory Barrier）：__sync_synchronize()，防止编译器和 CPU 重排。
