# 运输层（Transport Layer）

---

## 一. 运输层基础

### 1.1 运输层的定位与职责

在 TCP/IP 五层体系结构中，运输层（Transport Layer）位于应用层与网络层之间：

| 层次 | 英文 | 备注 |
|---|---|---|
| 应用层 | Application | 进程 |
| 运输层 | Transport | 本节重点 |
| 网络层 | Network / IP | 主机间寻址 |
| 数据链路层 | Link | 负责链路级传输 |
| 物理层 | Physical | 负责比特传输 |

运输层协议**运行在端系统（End Systems）**之中，而非中间路由器，其核心职责是在不同主机上运行的**应用进程之间**提供**逻辑通信（Logical Communication）**。

**发送端行为**：将应用层递交的报文（Message）切分为若干**报文段（Segment）**，添加运输层首部，向下递交给网络层。

**接收端行为**：从网络层接收数据报（Datagram）中的报文段，去除首部后将数据重组，向上递交给正确的应用进程。

> ⚠️ 注意区分：**网络层**负责主机（Host）之间的逻辑通信；**运输层**负责进程（Process）之间的逻辑通信。这一层次划分是运输层存在的根本原因。

---

### 1.2 为什么需要运输层

IP 层作为网络层协议，存在两大根本性局限，使得应用层无法直接依赖它进行可靠的进程间通信：

#### 局限一：寻址粒度不足（Addressing Granularity）

IP 数据报的目的地址只能定位到**主机（Host）**，但真实的端到端通信是发生在**应用进程（Application Process）**之间的。一台主机上可能同时运行着数十个网络应用（HTTP 服务、DNS 解析器、邮件客户端……），IP 层无法区分数据属于哪个进程。

而运输层可以通过**多路复用/解复用（Multiplexing / Demultiplexing）**机制，基于**端口号（Port Number）**解决了这一问题。

#### 局限二：服务模型过于薄弱（Weak Service Model）

IP 采用**尽力而为（Best-Effort）**的服务语义，对上层提供零可靠性保证：

| 故障类型 | 典型成因 |
|---|---|
| **数据包损坏（Corruption）** | 物理链路噪声、比特翻转 |
| **数据包丢失（Loss）** | 路由器队列溢出、主动丢包（RED 等 AQM 策略） |
| **数据包延迟（Delay）** | 网络拥塞、路由器排队等待 |
| **数据包乱序（Reordering）** | 多路径路由（ECMP）导致各路径时延差异 |
| **数据包重复（Duplication）** | 链路层重传或路由环路 |
| **无流量指导** | IP 层不限制发送速率，可能淹没网络或接收方 |

对应用开发者而言，直接在 IP 之上处理上述所有故障极为繁琐且容易出错。**运输层将这些复杂性封装在内，向应用层提供统一的、更高质量的服务抽象**。

---

### 1.3 运输层服务模型

Internet 中最核心的两个运输层协议为 **TCP** 和 **UDP**，二者在服务模型上存在本质差异：

| 特性 | TCP | UDP |
|---|---|---|
| **全称** | Transmission Control Protocol | User Datagram Protocol |
| **连接** | 面向连接（Connection-Oriented） | 无连接（Connectionless） |
| **可靠性** | 可靠、有序字节流（Reliable, In-Order Byte Stream） | 不可靠（Best-Effort Delivery） |
| **流量控制** | ✅ 有（Flow Control） | ❌ 无 |
| **拥塞控制** | ✅ 有（Congestion Control） | ❌ 无 |
| **性能保证** | ❌ 无时延/带宽保证 | ❌ 无 |
| **复用/解复用** | ✅ | ✅ |
| **Socket 类型** | SOCK_STREAM | SOCK_DGRAM |
| **定位** | 功能丰富，适合对可靠性要求高的应用 | 极简协议，仅提供复用/解复用能力 |

除 TCP/UDP 外，还有 SCTP（Stream Control Transmission Protocol）、MPTCP（多路径 TCP）、SST、RDP、DCCP 等协议，各有其特定的应用场景与设计权衡。

---

### 1.4 多路复用与解复用（Multiplexing & Demultiplexing）

多路复用（Mux）与解复用（Demux）是运输层最基础的核心功能，解决了"数据属于哪个进程"的问题。

#### 定义

| 术语 | 作用端 | 详细定义 |
|---|---|---|
| **多路复用（Multiplexing）** | 发送端 | 从多个不同的应用 Socket 收集数据块，分别封装运输层首部（其中包含端口信息），统一向下交付给网络层 |
| **解复用（Demultiplexing）** | 接收端 | 从网络层接收到的多路复用数据流中，依据运输层首部中的端口信息，将各报文段精确投递到对应的目标 Socket |

#### 工作示意图

```
        发送端                                     接收端
┌────────────────────┐                   ┌────────────────────┐
│  App P1  │  App P2 │                   │  App P3  │  App P4 │
└────┬─────┴────┬────┘                   └────┬─────┴────┬────┘
     │          │                             │          │
  Socket1    Socket2                       Socket3    Socket4
     │          │                             │          │
     └─────┬────┘                             └────┬─────┘
       多路复用（Mux）                         解复用（Demux）
           │ 添加运输层首部                          │  依据头部字段投递
      ┌─────────┐                             ┌─────────┐
      │  运输层  │ ---- 网络(IP 层) ----->      │  运输层  │
      └─────────┘                             └─────────┘
```

#### 数据段格式（TCP/UDP 通用）

```
<---------------- 32 bits ---------------->
┌─────────────────┬────────────────────┐
│  Source Port #  │    Dest Port #     │ <- 解复用的关键字段
├─────────────────┴────────────────────┤
│           Other Header Fields        │
├──────────────────────────────────────┤
│                                      │
│        Application Data (Payload)    │
│                                      │
└──────────────────────────────────────┘
```

主机接收到 IP 数据报后，从中提取运输层报文段，再结合 **IP 地址 + 端口号**共同决定将数据投递到哪个 Socket。

---

### 1.5 端口与 Socket

#### 端口号（Port Number）

- 端口号是 **16 位无符号整数**，范围 0～65535，用于区分同一主机上的不同应用进程
- 报文段的运输层首部中携带**源端口号（Source Port）**和**目的端口号（Dest Port）**

端口分类：

| 类型 | 范围 | 说明 |
|---|---|---|
| **周知端口（Well-Known Ports）** | 0 ～ 1023 | 由 IANA 统一分配，对应标准服务（HTTP:80、HTTPS:443、DNS:53、FTP:21、SMTP:25） |
| **注册端口（Registered Ports）** | 1024 ～ 49151 | 供应用程序注册使用 |
| **临时端口（Ephemeral Ports）** | 49152 ～ 65535 | 由操作系统动态分配给客户端，连接结束即释放 |

#### Socket

**Socket** 是应用进程与操作系统（运输层）之间交换网络消息的**软件抽象接口**，可理解为应用进程访问网络的"门"。

运输层寻址单元（Transport Layer Address）：`<主机 IP, 端口号>` 二元组，称为一个 **Socket 地址**。

操作系统维护端口与 Socket 之间的映射关系：

| Socket 类型 | 系统内核存储的映射键 |
|---|---|
| **UDP Socket**（SOCK_DGRAM） | `(本地端口, 本地 IP)` ↔ Socket |
| **TCP Socket**（SOCK_STREAM） | `(本地端口, 本地 IP, 远端端口, 远端 IP)` ↔ Socket |

---

### 1.6 无连接解复用（Connectionless Demultiplexing，UDP）

**核心规则**：UDP 解复用仅依赖**目的端口号（Dest Port）**，忽略源 IP 和源端口。

#### 创建与绑定

```java
// 服务端：绑定本地端口
DatagramSocket serverSocket = new DatagramSocket(6428);

// 客户端：创建 Socket（操作系统自动分配临时端口，如 9157）
DatagramSocket mySocket = new DatagramSocket(9157);
```

发送 UDP 数据报时必须显式指定：
- **目的 IP 地址（Destination IP Address）**
- **目的端口号（Destination Port Number）**

#### 解复用规则

当主机收到 UDP 报文段时：
1. 检查报文段中的**目的端口号**
2. 将报文段导向绑定了该端口号的 Socket

**关键推论**：来自不同源 IP / 源端口，但目的端口相同的 UDP 数据报，会被投递到**同一个 Socket**。

```
来源 A:9157 --(dst:6428)--\
来源 B:5775 --(dst:6428)---+----> serverSocket(6428)
来源 C:3001 --(dst:6428)--/
目的端口均为 6428，因此三者进入同一 Socket
```

#### 示例

```
mySocket2(9157)        serverSocket(6428)        mySocket1(5775)
      |                       |                       |
      |-- [9157 -> 6428] ---->|                       |
      |<-- [6428 -> 9157] ----|                       |
      |                       |<--- [5775 -> 6428] ---|
      |                       |---- [6428 -> 5775] -->|
```

> 没有链接的概念，因为所有的数据到达server之后会发网同一个sockets

---

### 1.7 面向连接解复用（Connection-Oriented Demultiplexing，TCP）

**核心规则**：TCP Socket 由**四元组（4-Tuple）**唯一标识，解复用时使用全部四个字段。

#### TCP Socket 的四元组标识

```
(源 IP, 源端口, 目的 IP, 目的端口)        
(Src IP, Src Port, Dst IP, Dst Port)   
```

接收端使用**全部四个值**来确定将到来的报文段导向哪个 Socket。

#### 与 UDP 解复用的对比

| 维度 | UDP | TCP |
|---|---|---|
| 解复用依据 | 仅目的端口 | 完整四元组 |
| 不同来源客户端 | 到同一 Socket | 到不同 Socket |
| 每客户端 Socket 数 | 1（共享） | 1（独占） |

#### 多客户端连接示意

```
客户端 A:9157 ----> Socket_PA（服务器 B:80，进程 P4）
客户端 C:5775 ----> Socket_PB（服务器 B:80，进程 P5）
客户端 C:9157 ----> Socket_PC（服务器 B:80，进程 P6）
```

三个报文段目的地均为 `(IP:B, Port:80)`，但由于源 IP / 源端口不同，四元组各不相同，因此被解复用到**三个独立的 Socket**，对应服务器上三个独立的处理进程/线程。

> 这才是真正的e2e!  

#### 服务器的并发处理模型

- **多进程/多线程服务器**：服务器为每个连接创建独立 Socket，对应独立进程或线程（Threaded Server），各 Socket 拥有其唯一的四元组。
- **非持久 HTTP（Non-Persistent HTTP）**：每次 HTTP 请求均创建新的 TCP 连接，因此每次请求对应一个不同的 Socket，请求完成即销毁。

## 二. 可靠传输的设计

### 2.1 理想信道 vs. 现实信道

#### 理想情况（Perfect Channel）

若信道完全无损、无延迟、无乱序：

```
发送方                                 接收方
  |                                      |
  |---- 数据包 -------------------------->|  直接收到并交付
  |---- 下一包 -------------------------->|
  |---- ...
```

在理想信道上，可靠传输**不需要任何额外机制**，发了就等、收了就用。

#### 现实情况（Best-Effort IP Network）

现实网络中，IP 层可能对数据包施加以下各种"伤害"：

```
网络层可能引入的故障：
  比特翻转   -> 损坏（Corrupted）
  队列溢出   -> 丢失（Lost）
  拥塞排队   -> 延迟（Delayed）
  多路径路由 -> 路径 1 / 路径 2 -> 乱序（Reordered）
  重传叠加   -> 重复（Duplicated）
```

可靠传输协议的目标：**在不可靠的网络层之上，为上层应用提供可靠的数据传输服务**。

---

### 2.2 可靠传输的应对机制全景

| 机制 | 针对的故障 | 详细说明 |
|---|---|---|
| **校验和（Checksum）** | 数据损坏 | 接收方对收到的报文段计算校验和，与首部中的校验和字段对比，检测传输错误 |
| **确认（ACK，Positive Acknowledgement）** | 损坏 / 丢失 | 接收方向发送方明确反馈：某个（或某些）数据包已被正确接收 |
| **否定确认（NACK，Negative Acknowledgement）** | 数据损坏 | 接收方向发送方明确反馈：某个数据包收到，但校验失败，请重传 |
| **序列号（Sequence Number）** | 乱序 / 重复 | 为每个数据包分配唯一编号，接收方据此识别重复包并重组有序数据流 |
| **重传（Retransmission）** | 丢失 / 损坏 | 发送方对未被确认的数据包进行重新发送 |
| **计时器（Timer）** | 丢失 | 发送方为每个已发但未确认的包设置超时计时器，超时即判定丢失并触发重传 |
| **前向纠错（FEC，Forward Error Correction）** | 数据损坏 | 发送方附加冗余校验信息，接收方在不重传的前提下自行纠正一定数量的错误 |
| **网络编码（Network Coding）** | 丢失 / 损坏 | 通过对多个数据包进行线性编码，更高效地修复多包丢失场景。目前的技术容错率只能做到30%左右。 |

---

### 2.3 处理数据包损坏

#### 基本方案：ACK / NACK + 重传

```
发送方                                 接收方
  |                                      |
  |---- P(1) --------------------------->|  ✓ 校验通过
  |<--------------------------- ACK -----|
  |                                      |
  |---- P(2) --------------------------->|  ✗ 校验失败
  |<-------------------------- NACK -----|
  |                                      |
  |---- P(2) --------------------------->|  ✓ 重传后校验通过
  |<--------------------------- ACK -----|
```

#### 深层问题：ACK/NACK 本身也可能在传输中损坏！

若 ACK 在返回途中发生比特错误，发送方收到的是一个**无法识别的反馈**，陷入两难：

- 若假设是 NACK → 重传 P(1) → 若接收方已正确收到 P(1)，重传的 P(1) 就是**重复包**
- 若假设是 ACK → 继续发 P(2) → 若实际是 NACK，则 P(1) **永久丢失**

**根本解决方案：为数据包和 ACK 均引入序列号（Sequence Number）**

```
发送方                                 接收方
  |                                      |
  |---- P(1) --------------------------->|  ✓ 收到新包 P(1)
  |<------------------------ ack(1) -----|
       [ack(1) 在途中损坏，发送方无法识别]
  |                                      |
  |---- P(1) --------------------------->|  识别为重复包，丢弃
  |<------------------------ ack(1) -----|  重发 ACK
  |                                      |
  |---- P(2) --------------------------->|
```

序列号使接收方能够区分"这是新包"还是"这是重传包"，从而正确处理重复数据。

---

### 2.4 处理数据包丢失

#### 方案：计时器驱动的损失检测（Timer-Driven Loss Detection）

**核心思想**：发包时启动计时器，若在超时（Timeout）时间内未收到 ACK，则判定包丢失并重传。

#### 场景一：数据包本身丢失

```
发送方                                 接收方
  |                                      |
  |---- P(1) ----------------------------|  x P(1) 在网络中丢失
                  [Timeout]
  |                                      |
  |---- P(1) --------------------------->|  ✓ 重传成功到达
  |<------------------------ ack(1) -----|
  |                                      |
  |---- P(2) --------------------------->|
```

#### 场景二：ACK 丢失（数据包成功送达，但确认丢失）

```
发送方                                 接收方
  |                                      |
  |---- P(1) --------------------------->|  ✓ 成功收到
  |<------------------------ ack(1) -----|  x ACK 在途中丢失
                  [Timeout]
  |                                      |
  |---- P(1) --------------------------->|  ⚠ 收到重复的 P(1)
  |<------------------------ ack(1) -----|  识别重复后丢弃，并重发 ACK
  |---- P(2) --------------------------->|
```

#### 场景三：超时设置过短，导致"伪超时"（Premature Timeout）

```
发送方                                 接收方
  |                                      |
  |---- P(1) -----------------------...    原始包仍在路上
       [Premature Timeout: 超时设置过短]
  |                                      |
  |---- P(1) --------------------------->|  原始包与重传包最终都到达
  |<------------------------ ack(1) -----|  首次返回 ACK
  |                                      |  后到达的副本被识别为重复包并丢弃
  |<------------------------ ack(1) -----|  再次返回 ACK
  |---- P(2) --------------------------->|
```

> **结论**：计时器超时触发重传，必然引入潜在的数据包重复问题。序列号机制是解决重复问题的唯一手段，二者相互依赖，缺一不可。

---

### 2.5 核心组件小结

通过以上分析，可靠传输所需的核心组件归纳如下：

| 核心组件 | 英文 | 作用 |
|---|---|---|
| 校验和 | Checksum | 检测比特错误（损坏检测） |
| 计时器 | Timer | 检测数据包丢失，超时触发重传 |
| 确认 | ACK / NACK | 接收方向发送方反馈接收状态，提高传输效率 |
| 序列号 | Sequence Number | 标识数据包身份，处理重复与乱序 |

## 三. 可靠传输协议的构造

### 3.1 停等协议（Stop-and-Wait）

停等协议（Stop-and-Wait Protocol）是将上述四大组件组合在一起的**最简单、正确的**可靠传输协议实现。

#### 协议逻辑

**发送方（@Sender）**：
```
发送 packet(I)
(重)启动计时器
等待反馈：
    若收到 ACK  -> I++，发送下一包
    若收到 NACK -> 重发 packet(I)
    若超时      -> 重发 packet(I)
（循环）
```

**接收方（@Receiver）**：
```
等待数据包
若收到的包校验正确 -> 发送 ACK
否则               -> 发送 NACK
（循环）
```

#### 协议特点

- **正确性（Correctness）**：停等协议是正确的——在有限次重传后，所有数据包均能被正确送达。
- **低效性（Inefficiency）**：发送方每次只允许一个包"在途（In-Flight）"，必须等收到 ACK 才能发下一包，导致链路大量空闲。

---

### 3.2 停等协议的效率分析

#### 时间空间示意

```
发送方                                 接收方
  |                                      |
  |---- 数据包 -------------------------->|  最后一个比特到达，耗时 L/R
  |                                      |  发出 ACK
  |<--------------------------- ACK -----|  ACK 到达，往返时延约为 RTT
  |---- 下一包 -------------------------->|
```

#### 发送方利用率公式

$$U_{sender} = \frac{L/R}{RTT + L/R}$$

其中：

- $L$：数据包大小（bit）
- $R$：链路带宽（bps）
- $L/R$：**传输时延（Transmission Delay）**，即将整个包注入链路所需时间
- $RTT$：**往返时延（Round-Trip Time）** = 2 × 传播时延

#### 数值计算示例

**参数设置**：

- 链路带宽：$R = 1 \text{ Gbps} = 10^9 \text{ bps}$
- 单向传播时延：15 ms → RTT = 30 ms
- 数据包大小：$L = 8000 \text{ bit}$

**传输时延**：

$$D_{trans} = \frac{L}{R} = \frac{8000}{10^9} = 8 \times 10^{-6} \text{ s} = 8 \; \mu\text{s}$$

**发送方利用率**：

$$U_{sender} = \frac{0.008 \text{ ms}}{30.008 \text{ ms}} \approx 0.00027 = 0.027\%$$

**实际吞吐量**：

$$TP = U_{sender} \times R = 0.00027 \times 1 \text{ Gbps} \approx 270 \text{ kbps} \approx 33 \text{ kB/s}$$

> **结论**：在 1 Gbps 链路上，停等协议实际吞吐量仅约 33 kB/s，**利用率不足 0.03%**。造成这一严重浪费的根本原因是：传输时延（8 μs）远小于 RTT（30 ms），链路在等待 ACK 的绝大部分时间内处于**空闲**状态。**网络协议的设计方式成为物理资源的瓶颈，而非物理资源本身。**

---

### 3.3 流水线协议（Pipelined Protocol）

#### 核心思想

**流水线（Pipelining）**：允许发送方在收到 ACK 之前，同时发送**多个尚未被确认**的"在途数据包（In-Flight Packets）"。

```
停等协议：
发送方                                 接收方
  |---- Pkt1 --------------------------->|
  |<--------------------------- ACK -----|
  |---- Pkt2 --------------------------->|

流水线协议（窗口 = 3）：
发送方                                 接收方
  |---- Pkt1 ---- Pkt2 ---- Pkt3 ------->|
  |<------------ ACK1 -- ACK2 -- ACK3 ---|
  |---- Pkt4 ---- Pkt5 ---- Pkt6 ------->|
```

#### 流水线对协议设计的影响

1. **序列号空间必须扩大**：需要能够区分多个同时在途的数据包
2. **发送方必须维护缓冲区**：缓存已发送但尚未被确认的包，以备重传
3. **接收方可能需要维护缓冲区**：接受并暂存乱序到达的包（取决于具体协议）

#### 流水线对利用率的提升

以 3 包流水线为例：

$$U_{sender} = \frac{3 \cdot L/R}{RTT + L/R} = \frac{0.024 \text{ ms}}{30.008 \text{ ms}} \approx 0.00081$$

利用率提升为停等协议的 **3 倍**。一般地，窗口大小为 n 时，利用率提升为停等协议的 **n 倍**（直至达到链路带宽上限）。

---

### 3.4 三个核心设计决策

流水线协议的具体实现需要回答三个核心问题：

| 核心决策 | 对应机制 |
|---|---|
| 发送方可以发哪些包？ | 滑动窗口机制（Sliding Window） |
| 接收方如何确认数据包？ | 累积 ACK（Cumulative ACK） / 选择 ACK（Selective ACK） |
| 发送方应重传哪些数据包？ | 回退 N 步（Go-Back-N, GBN） / 选择重传（Selective Repeat, SR） |

---

### 3.5 滑动窗口机制（Sliding Window）

#### 窗口的定义

**窗口（Window）**：由**相邻序列号**构成的集合，窗口大小（Window Size）设为 $n$，表示当前最多允许 $n$ 个数据包同时在途。

#### 发送方窗口

设 $A$ 为发送方**最后一个已被累积确认（无间隙）的包**序号，则发送方窗口为：

$$\text{Window}_{sender} = \{A+1,\ A+2,\ \ldots,\ A+n\}$$

```
序列号 ->
... [■][■][■][A+1][A+2][A+3][A+4][A+5][ ][ ][ ]...
              ^                   ^
              A+1              A+n（窗口右边界）
    <-已ACK-><------ 已发未 ACK ------><- 不可发 ->
      (暗色■)                            (空白□)
```

- **已 ACK（Already ACKed）**：序号 ≤ A 的包，已确认，可丢弃缓冲区副本
- **已发未 ACK（Sent but not ACKed）**：窗口内已发送、等待确认的包，须保留副本
- **不可发送（Cannot be sent）**：窗口右边界之外，尚不允许发送的包

#### 接收方窗口

设 $B$ 为接收方**最后一个无间隙连续收到**的包序号，则接收方窗口为：

$$\text{Window}_{receiver} = \{B+1,\ B+2,\ \ldots,\ B+n\}$$

```
序列号 ->
... [■][■][■][B+1][B+2][B+3][B+4][ ][ ]...
              ^              ^
              B+1           B+n
  <-已收ACK-> <---- 可接受未收 ----><-不可接收->
    (暗色■)                         (空白□)
```

- **已收 ACK（Received and ACKed）**：已顺序交付给应用层
- **可接受未收（Acceptable but not yet received）**：接收方愿意接受的包
- **不可接收（Cannot be received）**：超出窗口范围，即便到达也应丢弃

#### 窗口滑动机制

- 接收方：每当收到新包并发出 ACK，窗口向右滑动
- 发送方：每当收到新的 ACK（B 值增大），窗口向右滑动，允许发送新的包

这种"滑动"特性使得协议可以在不停止发送的前提下，持续推进数据传输，故名**滑动窗口**。

#### 滑动窗口的吞吐量上界

$$TP_{sliding} \approx \min\!\left(\frac{n \times DATA}{RTT},\ B_{link}\right)$$

与停等协议（相当于 $n=1$）的 $DATA/RTT$ 相比，滑动窗口将吞吐量提升了 $n$ 倍（在未达链路带宽上限时）。

> **当 $n$ 过大时**：发送速率超过链路容量或接收方处理能力，导致拥塞（Congestion）。这正是 TCP 拥塞控制机制发挥作用的场景。

---

### 3.6 确认机制：累积 ACK vs. 选择 ACK

#### 累积 ACK（Cumulative ACK）

- ACK 报文中携带的序号 = **接收方期望收到的下一个包的序号**
- 含义：该序号之前的所有包**均已被正确、有序地接收**
- 即 ACK($k$) 意味着序号 $1, 2, \ldots, k-1$ 的包全部已收到

**示例一**：顺序收到 B+1, B+2

```
收到 B+1, B+2 后：
  当前边界：B_new = B+2
  发送 ACK(B+3) = ACK(B_new + 1)
  窗口右移 2 格
```

**示例二**：只收到 B+4, B+5（中间 B+1, B+2, B+3 尚未到达）

```
只收到 B+4, B+5 时：
  当前状态：[...B][  ][  ][  ][ B+4 ][ B+5 ]...
                  ^ 中间空缺
  发送 ACK(B+1)
  仍期望 B+1，窗口不移动
```

#### 选择 ACK（Selective ACK，SACK）

- 对**每个正确收到的包单独进行确认**，无论其是否有序
- 提供更精细的接收信息，发送方可以只重传**真正丢失**的包
- 代价：簿记（Book-Keeping）更复杂，需要在 ACK 中携带更多信息

| 对比维度 | 累积 ACK | 选择 ACK |
|---|---|---|
| 信息精度 | 低（仅报告连续边界） | 高（精确报告每个包状态） |
| ACK 开销 | 小（只需一个序号） | 大（需要记录多个范围） |
| 实现复杂度 | 低 | 高 |
| 典型应用协议 | GBN 的标准实现 | SR / TCP SACK 选项 |

---

### 3.7 Go-Back-N 协议（GBN）

#### 协议规则

| 角色 | 规则 |
|---|---|
| **发送方** | 最多同时发送 $n$ 个未确认包；只为**第一个未确认包（A+1）**维护一个计时器 |
| **接收方** | 严格**只接受按序到达**的包（仅接受 B+1）；对所有乱序包**直接丢弃** |
| **确认方式** | 使用**累积 ACK**；ACK($k$) 中 $k$ = 下一期望接收的序号 |
| **超时处理** | 计时器超时后，**重传窗口内所有未确认包**：$A+1, A+2, \ldots, A+n$ |

#### 发送方窗口与接收方窗口的对应关系

```
发送方窗口 = {A+1, ..., A+n}   （n 个位置）
接收方窗口 = {B+1}             （始终只有 1 个位置！）
```

接收方"窗口大小"为 1，这是 GBN 的关键特征——接收方只期待下一个顺序包。

#### GBN 正常运行示例（窗口大小 = 3）

``` 
步骤  当前未确认分组  发送动作               接收方反馈
1     {1}             1 ----------------->   ACK 2
2     {1,2}           2 ----------------->   ACK 3
3     {1,2,3}         3 ----------------->   ACK 4
4     {2,3,4}         4 ----------------->   ACK 5
5     {3,4,5}         5 ----------------->   ACK 6
6     {4,5,6}         6 ----------------->   ACK 7
```

#### GBN 出错处理示例（窗口大小 = 3，包 5 丢失）

```
步骤  当前未确认分组  发送动作               接收方反馈
1     {1}             1 ----------------->   ACK 2
2     {1,2}           2 ----------------->   ACK 3
3     {1,2,3}         3 ----------------->   ACK 4
4     {2,3,4}         4 ----------------->   ACK 5
5     {3,4,5}         5 ----------------->   丢失
6     {4,5,6}         6 ----------------->   期待 5，丢弃 6，回 ACK 5
7     {5,6}           [Timeout: packet 5]
8     {5,6}           5 ----------------->   ACK 6
9     {6}             6 ----------------->   ACK 7
```

> **GBN 的代价**：一旦某包丢失，之后所有**已正确到达但乱序**的包全部被丢弃，超时后连同丢失的包一起重传。在错误率较高时，大量带宽被浪费在不必要的重传上。

---

### 3.8 Selective Repeat 协议（SR）

#### 协议规则

| 角色 | 规则 |
|---|---|
| **发送方** | 最多同时发送 $n$ 个未确认包；**为每个未确认包单独维护计时器** |
| **接收方** | 接受**窗口内所有正确到达的包**（包括乱序包）；将乱序包**缓存（Buffer）**在接收缓冲区中 |
| **确认方式** | 使用**选择 ACK**，对每个正确收到的包单独确认 |
| **超时处理** | 超时时**仅重传超时的那一个包** |

#### 发送方与接收方窗口

```
发送方窗口 = {A+1, ..., A+n}   （n 个位置）
接收方窗口 = {B+1, ..., B+n}   （n 个位置）
```

接收方也维护大小为 $n$ 的窗口，可以接收并缓存窗口内任意位置的包。

#### SR 出错处理示例（窗口大小 = 3，包 4 丢失）

```
步骤  发送窗口    发送动作               接收方行为
1     {1,2,3}     1 ----------------->   ACK=1
2                 2 ----------------->   ACK=2
3                 3 ----------------->   ACK=3
4     {2,3,4}     4 ----------------->   丢失
5     {3,4,5}     5 ----------------->   ACK=5，缓存并等待 4
6     {4,5,6}     6 ----------------->   ACK=6，缓存并等待 4
7                  [Timeout: packet 4]
8     {4,5,6}     4 ----------------->   ACK=4，随后顺序交付 4,5,6
9     {7,8,9}     7 ----------------->   继续发送
```

> **SR 的优势**：仅重传真正丢失的包（包 4），不重传已正确到达的包（5,6），**带宽利用率更高**。  
> **SR 的代价**：每个包需要独立计时器；接收方需维护接收缓冲区并管理乱序包；实现复杂度显著高于 GBN。

---

### 3.9 GBN vs. SR 综合对比

| 比较维度 | Go-Back-N（GBN） | Selective Repeat（SR） |
|---|---|---|
| **发送方窗口大小** | $n$ | $n$ |
| **接收方窗口大小** | 1（仅接受顺序包） | $n$（缓存乱序包） |
| **ACK 类型** | 累积 ACK（Cumulative） | 选择 ACK（Selective） |
| **计时器数量** | 1（针对最老未 ACK 包） | $n$（每个未确认包独立计时器） |
| **接收方缓冲区** | 不需要 | 需要（大小 = $n$） |
| **重传策略** | 超时后重传**整个窗口**内所有未确认包 | 超时后**仅重传**超时的那一个包 |
| **实现复杂度** | 较低 | 较高 |
| **带宽效率（低错误率）** | 高（重传开销小） | 高 |
| **带宽效率（高错误率）** | 低（大量不必要的重传） | 高（精准重传） |
| **适用场景** | 网络错误率低时更优 | 网络错误率高时更优 |

**选择原则**：

- 当网络**错误率低**时，GBN 简单且够用，SR 的复杂度得不偿失
- 当网络**错误率高**时，GBN 的"全窗口重传"浪费带宽，SR 的精准重传更高效

---

### 3.10 关键观察与总结

**观察一：链路充分利用**  
使用滑动窗口协议时，只要窗口大小 $n$ 足够大（即 $n \geq \lceil RTT \cdot R / L \rceil$），发送方就可以**持续不断地发包**，实现对链路带宽的充分利用。

**观察二：发送方缓冲区**  
发送方必须缓存**所有已发送但尚未被确认**的数据包副本，因为任何一个都可能需要重传。

**观察三：接收方缓冲区**  
接收方**可以**选择接收乱序到达的包（SR 要求这样做），但受限于接收缓冲区大小；GBN 接收方则选择直接丢弃乱序包，以换取实现简单性。

**观察四：协议复杂度的权衡**  
GBN 与 SR 是两种典范实现，现实中的 TCP 综合了二者的思想（使用累积 ACK 同时支持 SACK 选项），在两者之间取得工程上的平衡。

#### 可靠传输完整组件图

| 组件 | 英文 | 作用 |
|---|---|---|
| 校验和 | Checksum | 错误检测（Error Detection） |
| 计时器 | Timer | 丢包检测（Loss Detection） |
| 确认 | ACK / NACK | 支持累积确认（Cumulative）或选择确认（Selective） |
| 序列号 | SeqNo | 标识身份，处理重复与乱序 |
| 滑动窗口 | Sliding Window | 提升链路利用率（效率保障） |
| 可靠性协议 | GBN / SR | 综合以上组件，决定何时重传、确认什么 |
