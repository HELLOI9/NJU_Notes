# 同步基础

---

## 一、并发的核心困难

### 1.1 状态空间爆炸

并发程序的语句执行顺序具有**不确定性**。对于 $n$ 个线程各执行 $k$ 条语句的程序，其可能的交错执行顺序数量为 $\binom{nk}{k, k, \ldots, k}$，随线程数增加呈组合爆炸式增长，人类线性思维极难穷举。

以经典示例为例：

```c
int x = 0, y = 0;
void T1() { x = 1; int t = y; }
void T2() { y = 1; int t = x; }
```

两个线程各两条语句即产生 $\binom{4}{2} = 6$ 种合法交错顺序，且在**非顺序一致性（Non-Sequential Consistency）内存模型**下（如 TSO、PSO 等弱内存模型）还会有更多可能的执行结果（例如 `t1=0, t2=0` 在顺序一致性下不可达，但在 TSO 模型下可能出现）。

### 1.2 状态的分类

- **需要避免的状态**：如多个线程同时进入临界区（Critical Section），违背互斥性
- **需要到达的状态**：如 `pthread_join` 要求所有子线程终止后主线程才能继续执行

## 二、同步（Synchronization）

### 2.1 定义

**同步**是对并发的控制机制，使得"两个或两个以上随时间变化的量在变化过程中保持一定的相对关系"。

核心约束有两类：

- **不能太快**：线程到达某状态前需等待其他线程先完成某操作（如互斥）
- **不能太慢**：某条件成立后线程必须能够继续推进（活性保证）

> **互斥（Mutual Exclusion）是同步的一种特例**，而非同步的全部。

### 2.2 仅靠互斥无法完成同步

简单的 lock/unlock 只能保证**原子性**和**可见性**，无法控制线程的执行顺序。即便对每条语句加锁，线程的相对执行顺序仍然是任意的，无法保证特定的 happens-before 关系。

## 三、生产者-消费者问题（Producer-Consumer Problem）

### 3.1 问题描述

由 **Dijkstra** 首次提出，是并发领域的经典模型，能够代表绝大部分并发协调场景：有两类线程共享一个容量上限为 $n$ 的**有界缓冲区（Bounded Buffer）**，其中这两类线程分别是：

- **生产者**：若缓冲区有空位则放入数据，否则等待
- **消费者**：若缓冲区有数据则取走，否则等待

### 3.2 简化模型：打印括号
我们对生产者-消费者问题进行简化，规定生产 = 打印 `(`，消费 = 打印 `)`，缓冲区容量为括号嵌套深度上限 $n$.

于是有如下的合法性约束：

- 任意前缀中 `)` 的数量不超过 `(` 的数量（即深度 $\geq 0$）
- 任意时刻嵌套深度不超过 $n$（即深度 $\leq n$）

语义上等价于 `wait_until` 原语：

```c
void produce() { wait_until(depth < n) { printf("("); } }
void consume() { wait_until(depth > 0) { printf(")"); } }
```

### 3.3 错误尝试逐步分析

#### 尝试 0（无锁自旋）

```c
int n; //缓冲区大小
int depth = 0; //当前使用了多少空间

void produce() {
    while(1) {
    retry:
        int ready = (depth < n);//是否还有空间
        if (!ready) goto retry; //没有空间，自旋等待

        printf("("); //只有还有空间才能打印
        depth++; //打印之后，缓冲区使用的空间加一
    }
}
```

```c
void consume(){
    while(1){
    retry:
        int ready = (depth > 0);//是否有数据
        if(!ready) goto retry; //没有数据，自旋等待

        printf(")");//有数据才能打印
        depth--; //打印之后，缓冲区使用的空间减一
    }
}
```

**错误原因**：`depth` 是共享变量，很明显其读-判断-写构成**竞态条件（Race Condition）**，违背安全性。

#### 尝试 1（锁保护，但检查与操作分离）

```c
void produce() {
    while(1) {
    retry:
        mutex_lock(&lk);
        int ready = (depth < n);
        mutex_unlock(&lk);
        if (!ready) goto retry;

        mutex_lock(&lk);
        printf("(");
        depth++;
        mutex_unlock(&lk);
    }
}
```

```c
void consume(){
    while(1){
    retry:
        mutex_lock(&lk);
        int ready = (depth > 0);
        mutex_unlock(&lk);
        if (!ready) goto retry;

        mutex_lock(&lk);
        printf(")");
        depth--;
        mutex_unlock(&lk);
    }
}
```

**错误原因**：`unlock` 和后续操作之间存在**TOCTOU（Time-Of-Check-To-Time-Of-Use）漏洞**：因为虽然每次单独访问 `depth` 都是在锁内完成的，
但**“检查条件”和“执行修改”不是在同一次加锁里完成的，`ready` 为 `true` 时 `depth` 的值已经可能被其他线程修改。

#### 尝试 2（持锁自旋，正确但性能差）

```c
void produce() {
    while(1) {
    retry:
        mutex_lock(&lk);
        int ready = (depth < n);
        if (!ready){
            mutex_unlock(&lk);
            goto retry;
        }

        printf("(");
        depth++;
        mutex_unlock(&lk);
    }
}
```

```c
void consume(){
    while(1){
    retry:
        mutex_lock(&lk);
        int ready = (depth > 0);
        if (!ready) {
            mutex_unlock(&lk);
            goto retry;
        }

        printf(")");
        depth--;
        mutex_unlock(&lk);
    }
}
```

**正确性**：条件判断和操作均在锁的保护下原子完成，**安全性满足**。

**缺陷**：条件不满足时仍在反复获取、释放锁做**忙等待（Busy Waiting）**或**自旋（Spin）**，CPU 资源浪费，且其他线程被频繁唤醒后立即阻塞（惊群效应雏形）。

缺陷很好解决，可以利用系统的调度进行阻塞和唤醒。

#### 尝试 3（引入 wait/wakeup，但唤醒丢失）

```c
void produce() {
    while(1) {
    retry:
        mutex_lock(&lk);
        int ready = (depth < n);
        if (!ready){
            mutex_unlock(&lk);
            wait(&producer_waiting_list);
            goto retry;
        }

        printf("(");
        depth++;
        wakeup(&consumer_waiting_list);
        mutex_unlock(&lk);
    }
}
```

```c
void consume(){
    while(1){
    retry:
        mutex_lock(&lk);
        int ready = (depth > 0);
        if (!ready) {
            mutex_unlock(&lk);
            wait(&consumer_waiting_list);
            goto retry;
        }

        printf(")");
        depth--;
        wakeup(&producer_waiting_list);
        mutex_unlock(&lk);
    }
}
```

**错误原因**：`mutex_unlock` 和 `wait` 之间存在窗口期，若此时消费者已执行 `wakeup`，则该唤醒信号丢失，生产者永久沉睡（**Wakeup Lost / Lost Notification**）。

??? warning 注意
    “释放锁并进入等待状态”必须是原子的！

#### 尝试 4（futex 解决唤醒丢失，正确）

利用 `futex_wait(addr, val)` 的语义：若 `*addr == val` 才阻塞，否则立即返回，从而消除唤醒丢失窗口：

```c
int cnd1 = 0;   //用于唤醒生产者
int cnd2 = 0;   //用于唤醒消费者

void produce() {
    while(1) {
    retry:
        mutex_lock(&lk);
        int ready = (depth < n);
        if (!ready){
            int val = cnd1;
            mutex_unlock(&lk);
            futex_wait(&cnd1, val);
            goto retry;
        }

        printf("(");
        depth++;
        atomic_add(&cnd2, 1);
        futex_wake(&cnd2);
        mutex_unlock(&lk);
    }
}
```

```c
void consume(){
    while(1){
    retry:
        mutex_lock(&lk);
        int ready = (depth > 0);
        if (!ready) {
            int val = cnd2;
            mutex_unlock(&lk);
            futex_wait(&cnd2, val);
            goto retry;
        }

        printf(")");
        depth--;
        atomic_add(&cnd1, 1);
        futex_wake(&cnd1);
        mutex_unlock(&lk);
    }
}
```

上面的代码可以粗略理解成：只有当 `*cnd` 还等于我看到的旧值 `val` 时，才真正去睡；如果它已经变了，说明有线程改变过`depth`，此时`futex_wait()`直接返回，执行goto重新检查。

唤醒方通过 `atomic_add` 改变版本号再调用 `futex_wake`，保证唤醒不丢失。

??? info 注意
    为什么还要用`atomic_add(&cnd1, 1)`，明明已经有`lk`这把锁保护了？主要有两点原因：

    1. 如果以后有多个唤醒者，或者这段代码不再总是完全被同一把锁包住，就可能丢更新
    2. 遵循条件变量应该用原子方式更新的约定

## 四、条件变量（Condition Variable）

### 4.1 定义与语义

条件变量是上述 futex 模式的封装抽象，包含两个核心操作：

**`cond_wait(cond_t *cv, mutex_t *lk)`**

- 调用前提：调用线程已持有互斥锁 `lk`
- 行为：**原子地**释放锁并将自身阻塞挂起（两步不可分割，防止唤醒丢失）
- 返回时：重新获取锁 `lk`（此时条件不一定成立！）

**`cond_signal(cond_t *cv)`**

- 唤醒**一个**阻塞在 `cv` 上的线程
- 若无线程阻塞，则**什么也不做**（信号不储存，即无记忆性）

### 4.2 POSIX API

和互斥锁一样，Posix同样规定了相应的API：

```c
#include <pthread.h>
pthread_cond_t cv;
int pthread_cond_init(pthread_cond_t *cv, NULL);
int pthread_cond_destroy(pthread_cond_t *cv);
int pthread_cond_wait(pthread_cond_t *cv, pthread_mutex_t *mutex);
int pthread_cond_signal(pthread_cond_t *cv);
int pthread_cond_broadcast(pthread_cond_t *cv);
```

### 4.3 底层实现（基于 futex）

```c
typedef struct { 
    unsigned value; 
} cond_t;

void cond_wait(cond_t *cv, mutex_t *lk) {
    int val = atomic_load(&cv->value);
    mutex_unlock(lk);
    futex_wait(&cv->value, val);  // 原子检查后阻塞
    mutex_lock(lk);
}

void cond_signal(cond_t *cv) {
    atomic_fetch_add(&cv->value, 1);  // 版本号递增，防止丢失
    futex_wake(&cv->value);
}
```

> ⚠️ **易错点**：整数版本号存在溢出风险，实际生产实现（如 glibc 的 `pthread_cond_t`）使用更复杂的内部计数机制。

### 4.4 Signal 的两种语义

#### Hansen/Mesa 语义（非原子，主流实现）

- `cond_signal` 仅是一个"提示"，被唤醒线程重新竞争锁
- 唤醒与实际获得锁之间有时间窗口，其他线程可能抢先修改共享状态，被唤醒线程**不能保证**其等待的条件仍然成立！
- 实现简单，POSIX、Java `synchronized`/`Object.wait()` 均采用此语义，代言人：**Butler Lampson**（图灵奖得主，Mesa 系统）

#### Hoare 语义（原子，理论研究）

- `cond_signal` 将锁**原子地转移**给被唤醒线程，发起者自身阻塞
- 被唤醒线程结束或再次阻塞后，锁归还给发起者
- 被唤醒线程**保证**其等待的条件成立，程序性质易于证明，但实现复杂，调度器需额外承诺
- 代言人：**Tony Hoare**（图灵奖得主，Quick Sort、Hoare Logic 等）

### 4.5 Hansen/Mesa 语义下的正确范式

**黄金法则：`cond_wait` 必须在 `while` 循环中调用（而非 `if`）**。因为你不能认为你被唤醒之后条件一定成立。

```c
// 生产者
void produce() {
    while(1) {
        mutex_lock(&lk);
        while (!(depth < n))          // 用 while，不用 if
            cond_wait(&cv_p, &lk);
        printf("(");
        depth++;
        cond_signal(&cv_c);
        mutex_unlock(&lk);
    }
}

// 消费者
void consume() {
    while(1) {
        mutex_lock(&lk);
        while (!(depth > 0))          // 用 while，不用 if
            cond_wait(&cv_c, &lk);
        printf(")");
        depth--;
        cond_signal(&cv_p);
        mutex_unlock(&lk);
    }
}
```

### 4.6 为什么需要两个条件变量？

**使用单一条件变量 + `cond_signal` 可能导致死锁：**

反例：1个生产者 $P_1$，2个消费者 $C_1$、$C_2$，缓冲区容量为1：

1. $C_1$、$C_2$ 均因 `depth == 0` 阻塞
2. $P_1$ 生产1个数据，signal 唤醒 $C_1$，然后 $P_1$ 再次进入缓冲区满，阻塞
3. $C_1$ 消费数据后 signal，但此时 `cond_signal` 可能唤醒 $C_2$（同一条件变量）
4. $C_2$ 检查发现 `depth == 0`，重新阻塞
5. **结果**：$P_1$、$C_1$、$C_2$ 全部阻塞，系统死锁

**解决方案一：使用两个条件变量**（生产者等 `cv_p`，消费者等 `cv_c`，signal 时精确指向对方）

**解决方案二：使用 `cond_broadcast`**（唤醒所有线程，条件不满足者在 while 循环中重新睡眠）

```c
cond_broadcast(&cv);  // 替代 cond_signal，正确性保证但性能下降
```

### 4.7 条件变量使用经验法则（Rule of Thumb）

1. 必须有**共享状态变量**配合条件变量进行实际条件判断
2. 用**互斥锁保护**共享状态和条件变量操作
3. `wait`/`signal`/`broadcast` 必须在**持有互斥锁**时调用
4. 从 `wait` 返回后必须**重新检查条件**（`wait` 放在 `while` 循环中）
5. 不同条件使用**不同条件变量**（除非使用 `broadcast`）

## 五、条件变量的高级应用：基于计算依赖图的并行化

### 5.1 计算依赖图（Computation DAG）

定义有向无环图 $G = (V, E)$：

- $V$：计算事件（任务）集合
- $(u, v) \in E$：$u$ **happens-before** $v$，即 $v$ 依赖 $u$ 的计算结果

**并行化策略**：

- 无依赖路径的节点可并行执行
- 每个节点 $v$ 需等待所有 $\forall u: (u,v) \in E$ 的 $u$ 完成
- $u$ 完成后 signal 所有 $\forall v: (u,v) \in E$ 的 $v$

### 5.2 典型案例：Edit Distance 的并行 DP

$$dist(i,j) = \begin{cases} i & j = 0 \\ j & i = 0 \\ \min\{dist(i,j-1)+1,\ dist(i-1,j)+1,\ dist(i-1,j-1)+\mathbb{1}[A[i]\neq B[j]]\} & \text{otherwise} \end{cases}$$

$dist(i,j)$ 依赖 $dist(i-1,j)$、$dist(i,j-1)$、$dist(i-1,j-1)$，同一反对角线上的格子互无依赖，可**并行计算**。

### 5.3 通用线程池框架

实现思想：

1. （生产者）可以把调度的事情全部分给一个线程，而不是一次性把所有线程都直接放进内存，这个调度线程维护一个计算图，每次都会根据当前已经做完的计算机点，和依赖关系，调度可以运行的线程，把他们放进`ready`列表准备运行（这就是线程池）。这里的“生产者”更像任务投递者。
2. （消费者）被调度的运行的线程就执行计算，计算结束后通知调度线程，告知条件可能发生变化，可以重新计算下一批就可以计算的线程。这里的“消费者”更像任务执行者。

```c
void T_worker() {
    while (1) { 
        consume().run(); 
    }
}
void T_scheduler() {
    while (!jobs.empty()) {
        for (auto j : jobs.find_ready())
            produce(j);         // 将就绪任务放入队列
    }
}
```

**例：打印fish**
现有三个线程：

1. $T_a$：若干，无限循环打印<
2. $T_b$：若干，无限循环打印>
3. $T_c$：若干，无限循环打印_

任务：对线程同步，使得屏幕打印出<><_和><>_的组合。

解决方案：使用条件变量，回答三个问题：什么时候打印<，什么时候打印>，什么时候打印_？很自然的，我们想到了图灵机（有限状态机），剩下的就是套用同步的模版。

```c
#include <pthread.h>
#include <stdio.h>

typedef enum {
    S0,  // start
    S1,  // <
    S2,  // >
    S3,  // <>
    S4,  // ><
    S5,  // <><
    S6   // ><>
} State;

pthread_mutex_t lk = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t cv = PTHREAD_COND_INITIALIZER;

State state = S0;

int can_print_lt() {
    return state == S0 || state == S3;
}

int can_print_gt() {
    return state == S0 || state == S1 || state == S4;
}

int can_print_dash() {
    return state == S5 || state == S6;
}

void* Ta(void* arg) {
    while (1) {
        pthread_mutex_lock(&lk);
        while (!can_print_lt()) {
            pthread_cond_wait(&cv, &lk);
        }

        printf("<");
        fflush(stdout);

        if (state == S0) {
            state = S1;
        } else if (state == S3) {
            state = S5;
        }

        pthread_cond_broadcast(&cv);
        pthread_mutex_unlock(&lk);
    }
    return NULL;
}

void* Tb(void* arg) {
    while (1) {
        pthread_mutex_lock(&lk);
        while (!can_print_gt()) {
            pthread_cond_wait(&cv, &lk);
        }

        printf(">");
        fflush(stdout);

        if (state == S0) {
            state = S2;
        } else if (state == S1) {
            state = S3;
        } else if (state == S4) {
            state = S6;
        }

        pthread_cond_broadcast(&cv);
        pthread_mutex_unlock(&lk);
    }
    return NULL;
}

void* Tc(void* arg) {
    while (1) {
        pthread_mutex_lock(&lk);
        while (!can_print_dash()) {
            pthread_cond_wait(&cv, &lk);
        }

        printf("-");
        fflush(stdout);

        if (state == S5 || state == S6) {
            state = S0;
        }

        pthread_cond_broadcast(&cv);
        pthread_mutex_unlock(&lk);
    }
    return NULL;
}
```

## 六、信号量（Semaphore）

### 6.1 定义

由 **Dijkstra** 于 1965 年在 "THE Multiprogramming System" 中提出。信号量是一个**整数值**的同步原语，只能通过两个**原子**操作修改：

1. **`P(sem_t *sem)`**（prolaag，荷兰语"检验"；又称 down/wait/acquire/decrease）

    - 若 `sem->value > 0`：将值减一后返回
    - 若 `sem->value <= 0`：阻塞当前线程，加入等待队列

2. **`V(sem_t *sem)`**（verhoog，荷兰语"自增"；又称 up/post/signal/release/increase）

    - 将 `sem->value` 加一
    - 若有线程阻塞在该信号量上，唤醒其中一个

??? info 
    为什么需要“信号量”？

    条件变量是无记忆的，因此需要程序员手动进行额外的条件判定。这虽然很灵活，但是人类是不断追求简单的生物。一个很自然的需求就是能否有一个同步原语能够自带“状态”，然后根据预设好的约定，“状态”不满足就阻塞，否则就唤醒？有的有的！一个自带“整数”状态的同步原语就是信号量！

### 6.2 POSIX API

```c
#include <semaphore.h>
sem_t sem;
int sem_init(sem_t *sem, int pshared, unsigned int value);
int sem_destroy(sem_t *sem);
int sem_wait(sem_t *sem);   // P 操作
int sem_post(sem_t *sem);   // V 操作
```

### 6.3 基于互斥锁+条件变量的实现

```c
typedef struct {
    int value;
    pthread_cond_t cond;
    pthread_mutex_t lock;
} sem_t;

void sem_wait(sem_t *s) {
    mutex_lock(&s->lock);
    while (s->value <= 0)
        cond_wait(&s->cond, &s->lock);
    s->value--;
    mutex_unlock(&s->lock);
}

void sem_post(sem_t *s) {
    mutex_lock(&s->lock);
    s->value++;
    cond_signal(&s->cond);
    mutex_unlock(&s->lock);
}
```

### 6.4 信号量的初值语义（"资源数"视角）

| 初值 | 语义 | 应用场景 |
|------|------|----------|
| `1` | 二值信号量，等价于互斥锁 | 临界区互斥访问 |
| `0` | 阻塞语义，实现 happens-before | 线程顺序控制、join 等待 |
| `n > 1` | 计数信号量，表示 $n$ 份可用资源 | 有界缓冲区容量控制 |

### 6.5 典型用法

#### 实现互斥（二值信号量）
```c
sem_init(&sem, 0, 1);  // 初值为 1
void func() {
    P(&sem);
    // 临界区
    V(&sem);
    // 临界区
}
```

#### 实现 happens-before（初值为 0）

```c
sem_t sem;
int main(int argc, char *argv[]) {
    //set sem intial value to be X
    sem_init(sem_t &sem, 0, X);
    pthread_t c;
    create_thread(&c, child);
    // wait for thread
    P(&sem);
    do_something();
    return 0;
}

void* child(void *arg) {
    do_something();
    V(&sem); // signal here: child is done
    return NULL;
}
```

#### 解决生产者-消费者问题（简洁版）
```c
sem_t empty; sem_init(&empty, 0, MAX);  // 空位数
sem_t full;  sem_init(&full,  0, 0);    // 数据数

void produce() {
    while(1) { 
        P(&empty); 
        printf("(");
        V(&full); 
    }
}
void consume() {
    while(1) { 
        P(&full); 
        printf(")"); 
        V(&empty); 
    }
}
```

> ⚠️ **关键注意**：P/V 操作内部已释放互斥锁，P/V 之后的代码**不受锁保护**。若涉及共享缓冲区的实际读写，必须额外加互斥锁（上面的代码由于printf是天生线程安全的，所以没有这个问题）：

#### 正确的带共享缓冲区版本（避免死锁的关键顺序）
```c
// 正确：先 P 条件信号量，再 P 互斥信号量
sem_t mutex;  //用于互斥的信号量（也可以用互斥锁）
sem_init(sem_t &mutex, 0, 1);
sem_t empty;  //缓冲区有多少空
sem_init(sem_t &empty, 0, MAX);
sem_t full;   //缓冲区有多少资源
sem_init(sem_t &full, 0, 0);

void produce() {
    while(1) {
        P(&empty);
        P(&mutex);
        produce_shared_buffer();
        V(&mutex);
        V(&full);
    }
}

void consume() {
    while(1) {
        P(&full);
        P(&mutex);
        consume_shared_buffer();
        V(&mutex);
        V(&empty);
    }
}
```

> ⚠️ **死锁易错点**：若将互斥锁 P 操作放在条件信号量 P 之前（先锁再判断条件），当条件不满足时线程持锁阻塞，其他线程无法进入临界区改变条件，造成**死锁**。必须保证：**条件允许，才进入临界区**。另外，一般先`V(&mutex)`再`V(&empty)`或者`V(&full)`，这样可以确保先把临界区释放掉，保证被唤醒的线程更容易马上继续运行，不会刚醒又卡在`mutex`上太久。总结一下，互斥锁的使用规则就是“迟到早退”。

### 6.6 信号量的局限性

信号量**不擅长**表达以下场景：

1. **"二选一"的唤醒**：`V` 操作只能递增特定信号量，难以动态选择唤醒哪类线程（如打印 fish 问题中，完成一条鱼后下一个可以是 `<` 或 `>` 两种开头）
2. **非整数条件**：信号量状态仅为整数，无法表达任意谓词（如复合布尔条件）
3. **无 broadcast 机制**：标准信号量无法一次唤醒所有等待线程

总的来说，信号量主要是用于“资源处理”的问题，不适合布尔逻辑、状态枚举、有限状态机的问题。

## 七、条件变量 vs 信号量的对比

| 维度 | 条件变量 | 信号量 |
|------|----------|--------|
| **状态** | 无记忆（信号丢失不保留） | 有记忆（整数值累积） |
| **条件表达** | 任意谓词，灵活 | 仅整数计数 |
| **使用复杂度** | 需配合 mutex + 共享状态 | API 简洁，自带状态 |
| **精确唤醒** | signal/broadcast 均可 | V 操作具有指向性 |
| **适合场景** | 复杂条件同步（fish问题等） | 资源计数、简单ordering |
| **死锁风险** | 使用不当（if 而非 while）易出错 | 嵌套 P 操作顺序不当易死锁 |

## 八、常见易错点汇总

### 8.1 `cond_wait` 后用 `if` 而非 `while`

- **错误**：`if (!ready) cond_wait(...)` — 在 Mesa 语义下，唤醒后条件未必成立
- **正确**：`while (!ready) cond_wait(...)` — 始终重新检查条件

### 8.2 `cond_wait` 前未持有互斥锁

- 行为未定义，可能导致唤醒丢失或数据竞争

### 8.3 在 `cond_signal` 时未持有互斥锁

- 可能与 `cond_wait` 内部的 unlock-sleep 操作产生竞争，导致唤醒丢失

### 8.4 单一条件变量 + signal 导致唤醒错误线程

- 生产者唤醒了另一个等待同一 CV 的消费者（反之亦然），形成所有线程均阻塞的死锁
- 解决：使用两个 CV，或改用 `broadcast`

### 8.5 信号量 P 操作顺序导致死锁

- 不同线程以不同顺序对多个信号量执行 P 操作，形成循环等待
- 规则：所有线程必须以**相同顺序**获取多个信号量（类比锁的获取顺序）

### 8.6 条件变量的"无记忆"陷阱

- 若没有线程在等待，`cond_signal` 的信号**永久丢失**
- 必须配合共享状态变量，被唤醒后先检查条件，若已满足直接执行，无需依赖信号

### 8.7 信号量整数溢出

- 若 `V` 操作调用次数过多，信号量值可能溢出（虽然实际系统有保护，但在自实现时需注意）

### 8.8 `pthread_cond_t` 的虚假唤醒（Spurious Wakeup）

- POSIX 标准允许 `pthread_cond_wait` 在没有 signal/broadcast 的情况下自行返回
- 这是另一个必须用 `while` 而非 `if` 的理由
