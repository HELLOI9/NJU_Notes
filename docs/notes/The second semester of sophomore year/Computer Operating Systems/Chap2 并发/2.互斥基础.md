# 互斥(Mutual Exclusion)

---

## 一、并发基本术语

### 1.1 临界区（Critical Section）
访问**共享资源**的一段代码，该资源通常是一个变量或数据结构。

### 1.2 竞态条件（Race Condition）
出现在多个执行线程**大致同时**进入临界区时，它们都试图更新共享的数据结构，从而导致非预期的结果。

- 当程序含有一个或多个竞态条件时，程序的输出会因运行而异，具体取决于哪些线程在何时运行，因此结果是**不确定的（non-deterministic）**。

> **核心问题**：如何设计机制使得临界区避免出现竞态条件？

---

## 二、安全性（Safety）与活性（Liveness）

由 Leslie Lamport 提出，是运行程序（尤其是并发程序）的两个重要属性。

| 属性 | 含义 | 要求 | 示例 |
|------|------|------|------|
| **安全性（Safety）** | "没有坏事发生" | 执行中的**任何有限步骤内**都保持此性质 | 执行过程中不可出现除 0 错误 |
| **活性（Liveness）** | "好事终将发生" | 只要**最终**能满足要求即可，且执行中不能发生"不可挽回"的步骤 | 执行最终停止（中间出现无限循环就"不可挽回"） |

> **定理（Alpern and Schneider, 1987）**：程序执行上定义的任何性质，都是某个安全性性质与某个活性性质的组合。

---

## 三、临界区解决方案需满足的条件

### 3.1 互斥（Mutual Exclusion）— Safety
临界区内**最多只能有一个线程**同时存在。

### 3.2 行进（Progress）— Liveness
若当前临界区内没有线程，并且有线程想要进入临界区，那么**最终某个想要进入的线程会进入**该临界区。

### 3.3 有界等待（Bounded Waiting）— Fairness / No Starvation
若某个线程想要进入临界区，则其等待的期限有限（期间其他线程进入该临界区的次数有上限），不可一直排队等待。

- 若上限**未指定**具体数字：这是一个 **Liveness property**，线程最终会进入。
- 若上限**已指定**具体数字：这是一个 **Safety property**，因为在任何一次执行的有限步骤内，等待次数只要超过上限即发生了"坏事"。

### 3.4 性能（Performance）
进入和退出临界区的两个操作的**开销**，应相对于临界区内做的计算而言尽可能小。该性质刻画的是程序的 all the runs 的属性。

> **经验法则（Rule of Thumb）**：设计并发算法时，优先考虑安全性，但不要忘记活性和性能。

## 四、锁（Locks）

### 4.1 锁的定义
锁是一个**变量**，保存锁在某一时刻的状态：

- **可用的（available / unlocked / free）**：没有线程持有锁。
- **被占用的（acquired / locked / held）**：有一个线程持有锁，正处于临界区。

### 4.2 锁的两个配对操作

| 操作 | 语义 |
|------|------|
| `lock()` / `acquire()` | 尝试获取锁。若无其他线程持有锁，则获得锁并进入临界区；否则**阻塞**（不会返回，线程卡在那里）。 |
| `unlock()` / `release()` | 锁变为可用状态，若有因等待锁而阻塞的线程，其中一个将进入临界区。 |

锁为程序员提供了最小程度的调度控制，通过给临界区加锁保证互斥。

### 4.3 使用锁的代码示例

```c
// 示例一：原子累加
long sum = 0;

void *T_sum(void* nothing) {
    for (int i = 0; i < N; i++) {
        lock();
        sum++;        // 临界区
        unlock();
    }
}
```

```c
// 示例二：共享变量保护
int x = 0, y = 0;

void T1() {
    lock();
    x = 1; int t = y;  // Store(x); Load(y) — 临界区
    unlock();
    __sync_synchronize();  // memory barrier
    printf("%d", t);
}

void T2() {
    lock();
    y = 1; int t = x;  // Store(y); Load(x) — 临界区
    unlock();
    __sync_synchronize();
    printf("%d", t);
}
```

## 五、锁的实现尝试

### 5.1 尝试1：关中断（Disable Interrupt）

**思路**：单处理器系统中，程序被其他事件打断的机制仅有中断，关闭中断就可使当前程序状态机独占计算机系统。

| 架构 | 关中断指令 |
|------|-----------|
| x86 | `cli`（清除 eflags 中的 IF bit） |
| RISC-V | 清除 `mstatus` 的 MIE bit |
| ARM64 | `msr daifset, #3` |

- `lock()` → 关中断（disable interrupt）
- `unlock()` → 开中断（enable interrupt）

**问题**：

1. 临界区代码死循环 → 整个系统卡死
2. 中断关闭时间过长会导致重要的外界响应丢失（如磁盘 I/O 完成事件）
3. 关中断是**特权指令**，用户态应用无法执行，只有操作系统有权限（x86 可通过 cs 寄存器的最低两位判断特权级）
4. **多处理器无效**：每个处理器有独立的寄存器组，中断是每个处理器内部状态

> 注：在操作系统内核实现中，关闭中断是常见操作，但仅适用于单处理器内核场景。

#### 题外话：NMI（Non-Maskable Interrupts，不可屏蔽中断）

不是所有中断都可以被屏蔽，比如NMI：

- 主要用于处理**不可恢复的错误**（如内部芯片系统错误、系统数据损坏等）
- 若操作系统中断处理程序崩溃（死循环）且在关中断状态，NMI 可监控此错误并重启计算机
- **可行方案**：设置硬件定时触发 NMI，操作系统（正常运行下）定时复位定时器使 NMI 不被触发；一旦操作系统异常，timeout 会触发 NMI

---

### 5.2 尝试2：软件 Lock 标志

**思路**：使用一个标志变量来表达锁的状态（1 = 被占用，0 = 可用）。

```c
int flag = 0;

void lock() {
    while (flag == 1);  // 自旋观测 flag
    flag = 1;           // 设置
}

void unlock() {
    flag = 0;
}
```

> 注：此处假设 load 一个变量并测试（`flag == 1`）是原子的，设置一个变量（`flag = 1`、`flag = 0`）也是原子的。这两个假设在单处理器时代是合理的，但在现代计算系统已经不正确。

**问题（not safe）**：该方法只是把共享变量的竞态条件**转移到了 flag 这个状态变量上**。完全存在两个线程同时发现 flag 为 0，然后都进入临界区的可能。

```
时
间    Thread 1                Thread 2
      test $flag, 1
                              test $flag, 1
      set  $flag, 1
                              set  $flag, 1
      enter the critical section
                              enter the critical section   ← 同时进入！
```

**根本原因**：`test` 和 `set` 是两个**独立的指令**，没有原子的同时 test-and-set。

---

### 5.3 尝试3：互斥的 Test

**思路**：给每个线程的判定条件都不一样，且判定条件不可能同时对每个线程为 True。一个线程回答 "True" 意味着其他线程回答 "False"，在逻辑上就达成了互斥（严格轮转）。

```c
// for thread 0（此处用 flag == 0 作为阻塞条件）
void lock() {
    while (flag == 0);  // 自旋观测 flag
}
void unlock() {
    flag = 0;           // 设置
}

// for thread 1（此处用 flag == 1 作为阻塞条件）
void lock() {
    while (flag == 1);
}
void unlock() {
    flag = 1;
}
```

> 注：任何时刻两个线程的判定结果不同，这在内存顺序一致性（Sequential Consistency）模型下才成立。

**问题（违背 Liveness）**：一个线程能否获得锁，**完全依赖于另一个线程是否先进入了临界区**。

```c
void task() {
    while (1) {
        lock();
        critical_section_code();
        unlock();
    }
}
```

若 thread 1 始终不进入临界区，那么 thread 0 最多进入一次临界区后就无法再进入。这违背了行进（Progress）性质：当前没有线程在临界区，而 thread 0 想进入，但永远无法进入！

## 六、Peterson 算法

### 6.1 历史背景

20 世纪 60 年代，Dijkstra 向朋友们提出并发问题。数学家 **Theodorus Jozef Dekker** 首先给出解法——这是第一个被证明正确的"互斥"锁设计算法（但极为复杂）。**1981 年**，Gary L. Peterson 在论文 *"Myths about the mutual exclusion problem"* 中提出了一个更简单的算法。

### 6.2 核心思想

在尝试3的 test 基础上改进：除了测试 flag 之外，**还需要检查是否有其他线程想要进入临界区**（通过 `intents` 数组记录）。`flag` 的真正作用在于：当两个线程都有进入意图时，用于**解决冲突（轮转）**。

### 6.3 错误版本分析

下面是一个**不正确的**初步实现：

```c
int intents[2] = {0, 0};  // 进入意图
int flag = 0;             // 当前是谁可以获得锁（thread 0 or 1）
int self = i;             // 当前线程 ID，0 或 1

void lock() {
    intents[self] = 1;                      // 标记自己想要进入临界区
    while ((intents[(self + 1) % 2] == 1)  // 别人想不想进来
        && (flag == self));                 // 如果别人也想进来，看看是否轮到自己
}

void unlock() {
    intents[self] = 0;   // 标记自己不再想要进入临界区
    flag = self;         // ← 错误：flag 的修改放在这里会导致不安全
}
```

**反例（Unsafe）**：

```
|
|     Thread 1                              Thread 2
|     intents[0] = 0;（unlock 中）
|                                           enter the critical section（进入）
T                                           intents[1] = 0;
i                                           flag = 1
m     flag = 0                              
e     intents[0] = 1;
|     test intents[1] == 1 & flag == 0      intents[1] = 1;
|           -> false(access)                test intents[0] == 1 & flag == 1
|                                                -> false（access）
|     enter the critical section            enter the critical section   ← 同时进入！
```

flag 的设置放在 `unlock()` 中，导致冲突解决机制失效。**flag 应当在 `lock()` 中、设置意图之后立即设置**。

### 6.4 正确版本

```c
int intents[2] = {0, 0};  // 进入意图
int flag = 0;             // 当前是谁可以获得锁（thread 0 or 1）
int self = i;             // 当前线程 ID，0 或 1

void lock() {
    intents[self] = 1;                      // 标记自己想要进入临界区
    flag = self;                            // ← 关键：先表明意图，再设置轮转
    while ((intents[(self + 1) % 2] == 1)  // 别人想不想进来
        && (flag == self));                 // 如果别人也想进来，看看是否轮到自己
}

void unlock() {
    intents[self] = 0;   // 标记自己不再想要进入临界区
}
```

等价的完整形式（用于证明）：

```c
do {
    intents[i] = 1;                       // 语句 1
    flag = i;                             // 语句 2
    while ((intents[(i + 1) % 2] == 1)    // 语句 3（自旋等待）
        && (flag == i));
    critical_section();                   // 语句 4（临界区）
    intents[i] = 0;                       // 语句 5
    reminder_section();                   // 语句 6
} while (1);
```

## 七、Peterson 算法的正确性证明

### 7.1 状态表示

一个状态记为 `[t, h, k, f0, f1]`，其中：

- `t`：当前 `flag` 的值
- `h`：当前线程 T₁ 的语句 index
- `k`：当前线程 T₂ 的语句 index
- `f0`：`intents[0]` 的值
- `f1`：`intents[1]` 的值

### 7.2 Lemma 1（不变式）
当一个线程 Tᵢ 在调用 `lock()` 后和在离开临界区之前：`intents[self] = 1`

**证明**：trivial。只有调用 `unlock()`（语句 5）才会将 `intents[self]` 设为 0，而 `unlock()` 在临界区之后调用。

### 7.3 Lemma 2（安全性：互斥性）

**命题**：Peterson 算法能够保持互斥性（Safety）。

**证明（反证法）**：

不失一般性，假设状态 `[0, 4, 4, 1, 1]` 发生了，意味着 T₁ 和 T₂ 都进入了临界区（都在语句 4）。

由于 `flag = 0`，最后进入临界区的线程（记为最后通过自旋等待的线程）的判定语句一定为：

```
(intents[(i + 1) % 2] == 1) && (0 == i)
```

根据 Lemma 1，`intents[(i + 1) % 2] == 1` 这部分为 **True**，因此 `(0 == i)` 只能为 **False**，即 `i ≠ 0`，最后进入的是 **T₂**（i = 1）。

因此 `[0, 4, 4, 1, 1]` 的前一个状态是 `[0, 4, 3, 1, 1]`，即 T₂ 从状态 `[0, 4, 3, 1, 1]` 转移到进入临界区。

状态 `[0, 4, 3, 1, 1]` 的前驱有两种可能：

**前驱一：`[0, 3, 3, 1, 1]`**（T₁ 也在自旋等待）

然而若状态为 `[0, 3, 3, 1, 1]`，T₁ 的循环条件为：
```
(intents[1] == 1) && (flag == 0)
  →  true && true  = true
```
T₁ 的 while 条件为 True，T₁ 仍在自旋，**无法进入临界区**（语句 4）。矛盾！

**前驱二：`[?, 4, 2, 1, 1]`**（T₁ 早已进入临界区，T₂ 还在语句 2）

然而若状态为 `[?, 4, 2, 1, 1]`，T₂ 执行语句 2（`flag = 1`），这会将 `flag` 变为 **1**，不会有下一步的 `flag` 为 0。矛盾！

两种前驱均矛盾，假设不成立。由于对称性，**Peterson 算法是互斥的**。✓

```
[?, 4, 2, 1, 1]    [0, 3, 3, 1, 1]
        ↓                ↓
    [0, 4, 3, 1, 1]   ← 均矛盾
        ↓
    [0, 4, 4, 1, 1]   ← 假设的冲突状态
```

### 7.4 行进性（Progress）

**Peterson 算法能够保持行进性**：显然可以！

```c
while ((intents[(self + 1) % 2] == 1) && (flag == self));
//      ↑ 若没有其他线程想进入，此处直接为 false，while 不执行，直接返回
```

若当前临界区没有其他线程，则 `intents[(self + 1) % 2] == 0`，while 条件为 False，直接进入临界区。✓

### 7.5 有界等待（Bounded Waiting）

**Peterson 算法能够保持有界等待**：显然成立。

对于两个线程的情况，一个线程只要表达出进入临界区的意愿，**最多只需等一轮**。关键在于 `lock()` 中的 `flag = self`：在一个线程已经在等待进入的情况下，另一个线程无法连续进入两次（因为它 unlock 后再 lock 时会执行 `flag = self`，将轮转让给对方）。✓

## 八、模型检验（Model Checking）

由于手动证明繁琐，且对算法的微小改动需要重新证明，可以使用**模型检验（Model Checking）**自动化验证。

> **Model checking** is a method for formally verifying finite-state systems.

使用示例（`mosaic` 工具）：

```bash
python3 mosaic.py -c peterson.py | grep \"cs\" | sort | uniq
```

**局限性**：

1. 两个线程的模型检验遍历空间已经很大
2. 100 个线程或不指定上限的线程数时，状态空间爆炸

**相关研究**：

- *Model checking for programming languages using VeriSoft*
- *Finding and reproducing Heisenbugs in concurrent programs*
- *Using model checking to find serious file system errors*
- *VSync: Push-button verification and optimization for synchronization primitives on weak memory models*

## 九、回归现实：现代计算机架构的挑战

现实的 `load` 和 `store` 都**不是原子的**，甚至两个线程在同一时刻读取的同一变量值都可能不一致！

**原因**：现代多处理器系统中，每个处理器有私有缓存（Cache），处理器之间的内存视图不一致。Peterson 算法在现代计算机体系架构（多处理器 + 乱序指令流）下实际上是**错误的**。

> 尤其是“写读“乱序执行的影响！

### 9.1 内存屏障（Memory Barrier）

在现代体系架构下需要硬件支持的内存屏障指令：

| 架构 | 内存屏障指令 |
|------|------------|
| x86 | `mfence`（或 `lock` 前缀） |
| ARM | `dmb ish` |
| MIPS | `sync` |
| RISC-V | `fence rw, rw` |

GCC 编译器可以将 `__sync_synchronize()` 编译为对应架构的内存屏障。

但问题是：**这些 barrier 就够了吗？**——在某些弱内存模型（Weak Memory Model）下，仍然不足以保证 Peterson 算法的正确性。

## 十、硬件支持的锁

### 10.1 回归初心：原子化 Test-And-Set

**根本问题**：`test`（读取并判断）和 `set`（写入）是两条独立指令，需要将它们**原子化**成一条指令。

### 10.2 原子的 Test-And-Set（TAS）指令

X86 平台：使用 `lock` 前缀 + `cmpxchg` 指令实现 **Compare-And-Swap（CAS）**：

```
cmpxchg src, dst:
    if (dst == %ax) {
        dst = src;
        ZF = 1;
    } else {
        %ax = dst;
        ZF = 0;
    }
```

对应的 C 内联汇编实现：

```c
int flag;       // 需要被原子 test 和 set 的 flag
int expected = 0;

// 等价于：expected = __sync_val_compare_and_swap(&flag, 0, 1);
asm volatile (
    "lock cmpxchg %2, %1"
    : "+a" (expected)      // Value for comparison. x86 uses eax/rax.
    : "m"  (flag),         // Memory location
      "r"  (1)             // Value(1) to be written if flag == expected
    : "memory", "cc"
);
```

### 10.3 自旋锁（Spin Lock）的实现

有了 CAS 的硬件支持，自旋锁的实现如下（x86）：

```c
int flag = 0;

void lock() {
    int expected;
    do {
        expected = 0;
        asm volatile (
            "lock cmpxchg %2, %1"
            : "+a" (expected)
            : "m"  (flag),
              "r"  (1)
            : "memory", "cc"
        );
    } while (expected != 0);
}

void unlock() {
    asm volatile (
        "mov %1, %0"
        : "=m" (flag)
        : "r"  (0)
        : "memory"
    );
}
```

### 10.4 X86 硬件原子操作

所有 X86 CPU 都具有**锁定特定内存地址**的能力：当该地址被锁定后，阻止其他系统总线读取或修改该地址。

通过 `lock` 指令前缀 + 具体指令，即可在执行时"锁住"：

```
# 以下指令均可加 lock 前缀（X 开头的指令即使不加 lock 前缀也是原子的）
# （来源：IA-32 Intel® Architecture Software Developer's Manual）

ADD, ADC, AND, BTC, BTR, BTS,
CMPXCHG, CMPXCH8B, CMPXCHG16B,
DEC, INC, NEG, NOT, OR, SBB, SUB, XOR,
XADD, XCHG
```

### 10.5 Load-Linked / Store-Conditional（LL/SC）

除 CAS 之外，自旋锁也可以通过 **LL/SC** 机制实现，ARM、MIPS、RISC-V 均提供支持：

- **LL（Load-Linked）**：返回某内存位置的值，并标记该位置。
- **SC（Store-Conditional）**：仅当该内存位置自 LL 以来**未被更新**，写入才成功。

```
并发核心问题
├── 临界区（Critical Section）：访问共享资源的代码段
├── 竞态条件（Race Condition）：多线程并发访问导致的非确定性结果
│
├── 程序性质
│   ├── 安全性（Safety）："没有坏事发生" — 任何有限步内成立
│   └── 活性（Liveness）："好事终将发生" — 最终成立
│
├── 临界区解决方案要求
│   ├── 互斥（Mutual Exclusion）    → Safety
│   ├── 行进（Progress）            → Liveness
│   ├── 有界等待（Bounded Waiting） → Fairness
│   └── 性能（Performance）         → Overhead
│
└── 锁的实现演进
    ├── 尝试1：关中断        → 不适用多处理器，用户态无权
    ├── 尝试2：软件 flag     → test 与 set 非原子，不安全
    ├── 尝试3：严格轮转      → 违背行进性（Progress）
    ├── Peterson 算法        → 纯软件正确实现（顺序一致性模型）
    │   └── 现代架构下需要内存屏障（Memory Barrier）配合
    └── 硬件原子指令（TAS）
        ├── x86：lock + cmpxchg (CAS)
        └── ARM/MIPS/RISC-V：LL/SC
```
