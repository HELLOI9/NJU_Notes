# 互斥：进阶（Mutual Exclusion: Advances）

---

## 一、排号自旋锁（Ticket Lock）

### 1.1 问题背景：有界等待（Bounded Waiting）的缺失

普通自旋锁（spinlock）存在**有界等待**（Bounded Waiting）无法保证的问题：当一个线程在**自旋等待**（busy waiting / spin waiting）时，若其他线程持续竞争并不断被优先调度进入临界区，该线程可能**永远无法进入临界区**。

这违背了互斥算法的"有界等待"性质（即每个线程等待进入临界区的时间应有上界）。

> **解决思路**：引入"排队叫号"机制 —— 每个线程拿到一个唯一的票号，按票号顺序依次进入临界区，保证公平性（FIFO）。

---

### 1.2 数据结构定义

```c
typedef struct lock_ticket {
    int ticket;   // 当前发放的最大票号（下一个可领取的号）
    int turn;     // 当前应该进入临界区的票号
} lock_t;

lock_t flag;

void lock_init() {
    flag.ticket = 0;
    flag.turn   = 0;
}
```

**字段语义：**

| 字段 | 含义 |
|------|------|
| `ticket` | 全局递增的发号计数器，每次 lock 时原子地取走当前值并自增 |
| `turn` | 当前被允许进入临界区的票号，unlock 时自增 |

**不变量（Invariant）：** `flag.ticket - flag.turn` = 当前排队等待的线程数。

- 如果 `ticket == turn`：没人排队，锁空闲
- 如果 `ticket == turn + 1`：有一个线程正在持锁，没有额外等待者
- 如果 `ticket > turn + 1`：除了持锁线程，还有别的线程在排队

---

### 1.3 加锁与解锁实现（基于 `fetch_and_add`）

```c
void lock() {
    int myturn = 1;
    // atomic fetch-and-add
    // 等价于：myturn = __sync_fetch_and_add(&flag.ticket, 1);
    // 将 1 加到 flag.ticket，并将 flag.ticket 的"旧"值返回给 myturn
    asm volatile (
        "lock xaddl %0, %1"
        : "+r" (myturn), "+m" (flag.ticket)
        :
        : "memory", "cc"
    );
    while (flag.turn != myturn)
        ;   // spin
}

void unlock() {
    int value = 1;
    asm volatile (
        "lock xaddl %0, %1"
        : "+r" (value), "+m" (flag.turn)
        :
        : "memory", "cc"
    );
}
```

**关键点分析：**

- `lock xaddl`（locked exchange-add）是 x86 提供的**原子** fetch-and-add 指令，前缀 `lock` 保证总线级别的原子性。
- `lock()` 中：线程原子地从 `flag.ticket` 取走一个唯一号码 `myturn`，然后**自旋等待** `flag.turn == myturn`。
- `unlock()` 中：将 `flag.turn` 原子地加 1，唤醒下一个持有对应票号的线程。
- 由于每个线程持有唯一的票号，**有界等待**得以保证（每个线程最多等待 $N-1$ 个其他线程，$N$ 为总线程数）。

---

### 1.4 GCC 内建原子操作（Builtins）

GCC 提供了丰富的跨平台原子操作内建函数，避免手写内联汇编：

```c
// 原子地将 value 加到 *ptr，返回 *ptr 的旧值
type __sync_fetch_and_add(type *ptr, type value);

// 原子地比较 *ptr 与 oldval，若相等则将 *ptr 设为 newval，返回操作是否成功
bool __sync_bool_compare_and_swap(type *ptr, type oldval, type newval);

// 原子地将 *ptr 设为 value，返回 *ptr 的旧值（即 test-and-set）
type __sync_lock_test_and_set(type *ptr, type value);

// 原子地将 *ptr 置 0（释放锁）
void __sync_lock_release(type *ptr);
```

> **优势**：使用 GCC 内建函数可以编写**可跨平台**的原子操作，而不限于 x86 架构。  
> 更多参考：
> - https://gcc.gnu.org/onlinedocs/gcc/__sync-Builtins.html
> - https://gcc.gnu.org/onlinedocs/gcc/__atomic-Builtins.html

## 二、细粒度锁（Fine-Grained Lock）

### 2.1 "一把大锁"的问题

如果所有线程共用一把全局锁（Global Lock），则即使**彼此不相干**的线程也会被强制串行化，严重影响并发性能。

```c
// 粗粒度：全局锁（Global Lock）
void lock();
void unlock();
```

### 2.2 细粒度锁

为不同的共享资源分配独立的锁（Per-object Lock），只有竞争**同一资源**的线程才需要互斥：

```c
// 细粒度：锁作为参数传入（Per-object Lock）
typedef struct {
    ...
} lock_t;

void lock(lock_t *lk);
void unlock(lock_t *lk);
```

**好处**：不同资源的访问可以真正并行，显著提升多核系统的并发性能。

### 2.3 细粒度排号自旋锁实现

将全局 flag 改为参数指针形式：

```c
void lock(lock_t *flag) {
    int myturn = 1;
    // atomic fetch-and-add
    // 等价于：myturn = __sync_fetch_and_add(&flag->ticket, 1);
    asm volatile (
        "lock xaddl %0, %1"
        : "+r" (myturn), "+m" (flag->ticket)
        :
        : "memory", "cc"
    );
    while (flag->turn != myturn)
        ;   // spin
}

void unlock(lock_t *flag) {
    int value = 1;
    asm volatile (
        "lock xaddl %0, %1"
        : "+r" (value),
          "+m" (flag->turn)
        :
        : "memory", "cc"
    );
}
```

## 三、自旋锁的正确性前提

### 3.1 锁不是"魔法"保护

`lock()` / `unlock()` 本身**没有真正意义上地保护临界区的共享资源**。互斥锁的正确性完全依赖于所有线程的**协作**：

- 每一个访问共享资源的线程都必须在访问前调用 `lock()`，访问后调用 `unlock()`。
- **只要有一个线程绕过锁直接访问共享资源，整个互斥逻辑就完全失效。**

```c
// 正确的使用方式：
T1: spin_lock(&lk);   sum++;   spin_unlock(&lk);
T2: spin_lock(&lk);   sum++;   spin_unlock(&lk);

// 错误！T3 绕过了锁：
T3: sum++;   // ← 一个不正确的访问，整个逻辑就错了
```

> **结论**：互斥锁是一种**合约机制**（Convention），而非硬件强制隔离。程序员必须遵守"凡访问共享资源，必须持有对应的锁"这一约定。

## 四、内核中的自旋锁问题

### 4.1 内核使用自旋锁的场景

操作系统内核是资源的管理者，因此内核中存在大量共享资源的并发访问场景，互斥锁使用非常普遍，例如：

- 多个线程通过系统调用（如 `write`）访问共享的终端输出时，OS 使用锁互斥各线程。
- **中断处理程序（Interrupt Handler）**中也可能用到自旋锁（这是用户程序所没有的）。

### 4.2 中断引发的死锁（Interrupt-induced Deadlock）

**场景描述：**

1. 一个线程通过系统调用访问某共享变量，内核已对该变量**上锁**（持有锁）。
2. 此时一个**中断**（Interrupt）发生，CPU 被强制切换到**中断处理程序**（ISR）。
3. 该 ISR 同样需要访问同一共享变量，于是尝试**获取同一把锁**。
4. 但该锁已被系统调用程序持有，ISR 开始**自旋等待**。
5. 由于中断处理程序的优先级通常高于系统调用，系统调用程序**无法被恢复执行**。
6. **结果**：ISR 在等待一个永远不会被释放的锁 —— **死锁（Deadlock）**！

```
syscall starts → acquires lock → [Interrupt!] → ISR starts → tries to acquire lock → Spin! → 永远等待
```

### 4.3 解决方案：保存并恢复中断状态

**思路**：在获取自旋锁之前**关闭中断**，在释放锁时**恢复中断状态**（而非无条件开启）。

**错误的简单实现**：直接在 lock 前关中断、unlock 时开中断 —— 如果在此次自旋之前已处于关中断状态，unlock 时开中断就会破坏之前的中断状态，导致嵌套锁场景下的错误：

```
线程开始 → 关中断 → Lock(A) → 关中断(无效) → Lock(B) → Unlock(B) → 开中断(错！破坏了 Lock(A) 需要的关中断状态) → Unlock(A) → 开中断
```

**正确做法**：使用**计数器**记录中断被关闭的嵌套深度，只有当嵌套深度归零时才真正恢复中断。

### 4.4 xv6 中自旋锁的实现

xv6（MIT 开发的教学用类 Unix 操作系统）提供了一个经典的正确实现：

```c
// 自旋锁数据结构
typedef struct {
    const char   *name;   // 锁的名称（用于调试）
    int           status; // LOCKED / UNLOCKED
    struct cpu   *cpu;    // 当前持有该锁的 CPU
} spinlock_t;
```

```c
// 加锁
void spin_lock(spinlock_t *lk) {
    // 关中断，避免死锁（保存并记录之前的中断状态）
    push_off();

    // 检测同一 CPU 重复获取同一把锁（会导致死锁）
    if (holding(lk)) {
        panic("acquire %s", lk->name);
    }

    // 自旋等待，直到成功将状态从 UNLOCKED 切换为 LOCKED
    int got;
    do {
        got = atomic_xchg(&lk->status, LOCKED);
    } while (got != UNLOCKED);

    lk->cpu = mycpu;
}
```

```c
// 解锁
void spin_unlock(spinlock_t *lk) {
    if (!holding(lk)) {
        panic("release %s", lk->name);
    }

    lk->cpu = NULL;

    // 原子地将状态设置回 UNLOCKED
    atomic_xchg(&lk->status, UNLOCKED);

    // 恢复之前的中断状态
    pop_off();
}
```

```c
// push_off()：关闭中断，并记录嵌套深度和原始中断状态
void push_off(void) {
    int old = ienabled();    // 记录当前中断是否开启
    struct cpu *c = mycpu;

    iset(false);             // 关闭中断

    if (c->noff == 0) {
        c->intena = old;     // 只在第一次关中断时保存原始状态
    }
    c->noff += 1;            // 嵌套计数 +1
}

// pop_off()：恢复中断状态（仅当嵌套深度归零时才真正开中断）
void pop_off(void) {
    struct cpu *c = mycpu;

    // 断言：持有锁时中断必须是关闭的
    if (ienabled()) {
        panic("pop_off - interruptible");
    }
    if (c->noff < 1) {
        panic("pop_off");
    }

    c->noff -= 1;

    // 只有当所有嵌套锁都释放，且原始状态为开中断，才真正开中断
    if (c->noff == 0 && c->intena) {
        iset(true);
    }
}
```

**设计要点总结：**

| 机制 | 说明 |
|------|------|
| `push_off()` / `pop_off()` | 通过计数器 `noff` 实现中断开关的嵌套管理 |
| `intena` | 保存进入第一把锁之前的中断使能状态 |
| `noff` | 记录当前已关闭中断的嵌套层数（lock 深度） |
| 只有 `noff == 0` 时才真正开中断 | 防止内层 unlock 错误地开启外层 lock 所需的中断关闭状态 |

## 五、用户态互斥锁问题

### 5.1 自旋锁在用户态的性能问题

用户态程序中，临界区往往较长（与内核"短临界区"的常态不同），自旋锁存在严重的性能问题：

**问题一：CPU 空转浪费**

除了持有锁的线程，其他处理器上等待的线程都在做**忙等待**（busy-waiting），消耗 CPU 周期：

```
lock ─────────────── unlock
 A  [running in CS                    ]
 B  [spin spin spin spin spin ...     ] lock [running]
 C  [spin spin spin spin spin spin ...] 
 D  [spin spin spin spin spin spin ...]
                                            ▶ t
```

**问题二：中断切出后浪费更严重**

若持有锁的线程被中断切出（preempted），其他线程会在整个时间片都在自旋：

```
lock  interrupt               resumed   unlock
 A  [  ]─────────────────────────[         ]
 B  [spin spin spin spin spin spin ...     ] lock
 C  [spin spin spin spin spin spin spin ...]
 D  [spin spin spin spin spin spin spin ...]
                                              ▶ t
```

> 注意：用户态**无法通过关闭中断**来解决这个问题（关中断是特权操作）。

### 5.2 简单改进方案：`yield`

让无法获得锁的线程主动**让出 CPU**（yield），而不是空转：

```c
void yield_lock(spinlock_t *lk) {
    while (xchg(&lk->locked, 1)) {
        // sched_yield() 的封装：主动让出 CPU 给其他线程
        syscall(SYS_yield);   // yield() on AbstractMachine
    }
}

void yield_unlock(spinlock_t *lk) {
    xchg(&lk->locked, 0);
}
```

### 5.3 yield 方案的问题

`yield` 只是让线程**暂时**让出 CPU，该线程仍处于 **ready（就绪）** 状态，可随时被再次调度。

**问题**：在获得锁之前，线程反复经历"被调度 → yield 让出 → 被再次调度 → 发现锁仍被占用 → yield..."，会产生**大量不必要的上下文切换（Context Switch）开销**：

```
lock ─────────────────────────────────────────────── unlock
 A  [running in CS                                      ]
 B  [run][yield][run][yield][run][yield][run][yield][run] lock
 C  [run][yield]     [run][yield]     [run][yield]
 D       [run][yield]     [run][yield]     [run][yield]
                                                         ▶ t
                   └── unnecessary overhead ──┘
```

### 5.4 正确解决方案：OS 协助的 Sleep/Wakeup

用户态锁操作应与 OS 调度器**协同配合**，让无法获得锁的线程真正**阻塞**（blocked），不再参与调度：

**`mutex_lock(&lk)` 语义：** 尝试获取锁。若失败（锁已被持有），则通过系统调用将当前线程标记为**阻塞（blocked）**，加入该锁的**等待队列**，然后让出 CPU（此时线程不在就绪队列中，不会被调度）；若成功，则进入临界区。

**`mutex_unlock(&lk)` 语义：** 释放锁。若等待队列中有线程：通过系统调用**唤醒（wakeup）**一个线程，将其从等待队列移出，标记为**就绪（ready）**，加入就绪队列，可被重新调度。

**OS 需要**为每个锁维护一个与其关联的**等待队列**（Wait Queue）。

---

### 5.5 朴素实现的问题：丢失唤醒（Lost Wakeup）

下面的朴素实现存在竞态条件（Race Condition）：

```c
void mutex_lock(spinlock_t* lk) {
    int got;
    do {
        got = atomic_xchg(lk->status, LOCKED);
        if (got != UNLOCK) {
            // 将当前线程加入等待队列，并标记为阻塞，释放 CPU
            wait(lk->wait_list);
        } else {
            break;
        }
    } while (1);
}

void mutex_unlock(spinlock_t* lk) {
    atomic_xchg(lk->status, UNLOCKED);
    if (!is_empty(lk->wait_list)) {
        // 等待队列中的一个线程移出队列，标记为就绪
        wakeup(lk->wait_list);
    }
}
```

**问题场景（Lost Wakeup）：**

```
时间
Thread 1                              Thread 2
got != UNLOCK                         
(还未执行 wait，即还未加入等待队列)
                                      atomic_xchg(&lk->status, UNLOCKED);
                                      if (!is_empty(lk->wait_list)) { ... }
                                      // 此时等待队列为空，wakeup 丢失！
wait(lk->wait_list);
// Thread 1 永远沉睡，无人唤醒
```

> **根本原因**：`got != UNLOCK` 的判断 和 `wait()` 的执行之间存在**窗口期**，若 unlock 恰好在此期间发生，wakeup 信号就会丢失。

## 六、Linux Futex（Fast Userspace muTEX）

### 6.1 Futex 提供的两个系统调用

Linux 提供了 Futex（Fast Userspace muTEX）机制，通过两个系统调用解决上述问题：

1. **`futex_wait(int *address, int expected)`**：**原子地**检测 `*address` 的当前值是否等于 `expected`；

    1. **相等**：将当前线程阻塞，挂入内核等待队列；
    2. **不等**：立即返回（不阻塞），让用户线程立刻重试 lock。

    这里的**关键**是需要原子性地检查并阻塞，消除 Lost Wakeup 的时间窗口。

2. **`futex_wake(int *address)`**：唤醒一个正在等待 `address` 所指向锁的线程。

### 6.2 基于 Futex 的完整互斥锁实现

```c
#define UNLOCK   0   // 锁空闲
#define ONE_HOLD 1   // 锁被持有，无等待者
#define WAITERS  2   // 锁被持有，有等待者
```

```c
void mutex_lock(spinlock_t* lk) {
    // Fast Path：尝试原子地从 UNLOCK 切换到 ONE_HOLD
    // cmpxchg 返回旧值
    int c = cmpxchg(lk, UNLOCK, ONE_HOLD);

    if (c != UNLOCK) {
        // Slow Path：锁已被持有
        do {
            if (c == WAITERS
                // 若旧状态是 ONE_HOLD，尝试升级为 WAITERS（表示有等待者）
                // cmpxchg 失败（返回非0）说明状态已变化，需重新检查
                || cmpxchg(lk, ONE_HOLD, WAITERS) != 0) {
                // 原子地检查 *lk 是否仍为 WAITERS，是则阻塞
                futex_wait(lk, WAITERS);
            }
            // 重新尝试获取锁：从 UNLOCK 切换到 WAITERS（保留等待者标记）
        } while ((c = cmpxchg(lk, UNLOCK, WAITERS)) != 0);
    } else {
        return;   // Fast Path 成功！
    }
}
```

```c
void mutex_unlock(spinlock_t* lk) {
    // atomic_dec 原子地将 *lk 减 1，并返回减之前的旧值
    if (atomic_dec(lk) != ONE_HOLD) {
        // 减之前不是 ONE_HOLD，说明有等待者（WAITERS）
        lk = UNLOCK;          // 先将锁置为 UNLOCK
        futex_wake(lk);       // 再唤醒一个等待线程
    } else {
        return;   // Fast Path 解锁！无等待者，直接返回
    }
}
```

### 6.3 正确性分析

**为何不会出现 Lost Wakeup？**

- `mutex_unlock` 先将 `*lk` 置为 `UNLOCK`，再调用 `futex_wake`。
- 即使存在如下竞态：Thread 1 完成了 `cmpxchg(lk, ONE_HOLD, WAITERS)` 但还未调用 `futex_wait`，此时 Thread 2 执行了 unlock（将 `*lk` 置 0），Thread 1 随后调用 `futex_wait(lk, WAITERS)` 时，由于 `*lk` 当前为 0 ≠ WAITERS，`futex_wait` 会**立即返回**而非阻塞 —— 从而避免了 Lost Wakeup。

**为何保证互斥？**

- 任意时刻只有一个线程能将 `*lk` 从 `UNLOCK` 切换为 `ONE_HOLD`（cmpxchg 原子操作）。
- 其他线程若发现锁被持有，会将 `*lk` 升级为 `WAITERS` 并阻塞。
- unlock 只有在 `*lk` 变为 `UNLOCK` 后才会唤醒等待线程，被唤醒的线程还需再次竞争 cmpxchg 才能真正获得锁。

### 6.4 两阶段锁（Two-Phase Lock）

Linux 的 Futex 实现实质上是一种**两阶段锁**，综合了自旋锁和睡眠锁的优点：

| 阶段 | 名称 | 行为 | 适用场景 |
|------|------|------|----------|
| 第一阶段 | **Fast Path（快速路径）** | 一次原子指令尝试获取锁，若成功直接进入临界区，无需系统调用 | 无竞争或低竞争 |
| 第二阶段 | **Slow Path（慢速路径）** | 自旋失败后，调用 `futex_wait` 阻塞当前线程，让出 CPU | 高竞争 |

> 真实的 Futex 有更多优化和细节，参见：
> - *Futexes are Tricky* by Ulrich Drepper
> - *LWN: A futex overview and update*

### 6.5 系统调用加锁的开销

虽然睡眠锁避免了自旋浪费 CPU，但每次 Lock/Unlock 都可能需要**陷入内核**（syscall），带来额外开销（如上下文切换、内核/用户态切换）。

> **注意**：在真实 workload 中，多线程激烈竞争同一把锁的场景并不常见。多数情况下是单线程独占锁，此时使用自旋锁反而更快（无需陷入内核，也不会空转）。这正是 Futex 两阶段设计的意义所在。

### 6.6 POSIX 互斥锁（pthread_mutex）

POSIX 标准提供了封装良好的互斥锁 API（底层通常基于 Futex 实现），在有操作系统的环境下**推荐使用**：

```c
#include <pthread.h>

pthread_mutex_t mutex;

// 初始化互斥锁
int pthread_mutex_init(pthread_mutex_t *restrict mutex, NULL);

// 销毁互斥锁
int pthread_mutex_destroy(pthread_mutex_t *mutex);

// 加锁（阻塞式）
int pthread_mutex_lock(pthread_mutex_t *mutex);

// 解锁
int pthread_mutex_unlock(pthread_mutex_t *mutex);
```

## 七、并发数据结构

### 7.1 线程安全（Thread-Safe）数据结构

**定义**：可以被多个线程安全地**并发访问**的数据结构，也称为**并发数据结构**（Concurrent Data Structure）。

**实现要点**：在访问和修改共享数据结构时，需要使用锁（一把或多把）进行保护。

**最简单的实现：一把大锁（One Big Lock）**

- 所有访问操作都串行化。
- 早期 Linux 采用了这一策略（Big Kernel Lock，BKL）：**简单、正确！**
- **缺点**：在多处理器（Multi-Processor）时代，大锁成为性能瓶颈（无法充分利用多核并行性）。

### 7.2 并发 Counter 示例（One Big Lock 版）

```c
typedef struct __counter_t {
    int value;
    pthread_mutex_t lock;
} counter_t;

void init(counter_t *c) {
    c->value = 0;
    Pthread_mutex_init(&c->lock, NULL);
}

void increment(counter_t *c) {
    Pthread_mutex_lock(&c->lock);
    c->value++;
    Pthread_mutex_unlock(&c->lock);
}

void decrement(counter_t *c) {
    Pthread_mutex_lock(&c->lock);
    c->value--;
    Pthread_mutex_unlock(&c->lock);
}

int get(counter_t *c) {
    Pthread_mutex_lock(&c->lock);
    int rc = c->value;
    Pthread_mutex_unlock(&c->lock);
    return rc;
}
```

### 7.3 高性能 Counter：近似计数器（Approximate Counter）

**思路**：将竞争分散，减少对全局锁的争抢：

- 每个 CPU 维护一个**本地 counter**（per-CPU local counter），由本地锁保护（per-CPU lock）。
- 所有 CPU 共享一个**全局 counter**（global counter），由全局锁保护。
- 本地 counter 积累到一定阈值（threshold）后，才将值**批量提交**到全局 counter（并将本地 counter 清零）。

**示意图：**

```
时间    L1      L2      L3      L4      G
        0       0       0       0       0
        1       0       1       1       0
        2       0       2       1       0
        3       0       3       2       0
        4       1       3       3       0
      5→0       1       3       4       5  (from L1)
        0       2       4     5→0      10  (from L4)
```

**优势**：本地操作只需本地锁（无跨核竞争），极大提升并发性能；全局 counter 的同步频率可调（threshold 越大，并发性越好，但全局值精度越低）。

## 八、层级化的并发解决方案

并发控制的实现是一个**层次化**（Layered）的体系，各层相互配合：

```
┌─────────────────────────────────────────────────┐  用户态
│           Concurrent Programs                   │
├─────────────────────────────────────────────────┤
│    glibc(pthread_mutex_lock/unlock etc. API)    │  用户态/内核态边界
├─────────────────────────────────────────────────┤
│  Kernel concurrency primitives(futex_wait/wake) │  内核态
├─────────────────────────────────────────────────┤
│  Hardware atomic instructions(cmpxchg, xaddl)   │  硬件
└─────────────────────────────────────────────────┘
```

| 层次 | 提供方 | 代表性机制 |
|------|--------|-----------|
| 硬件层 | CPU / ISA | `cmpxchg`、`xaddl`、`lock` 前缀 |
| 内核层 | 操作系统内核 | `futex_wait`、`futex_wake`、`push_off`/`pop_off` |
| 库函数层 | glibc / POSIX | `pthread_mutex_lock`、`pthread_mutex_unlock` |
| 应用层 | 用户程序 | 调用上层 API 构建正确、高效的并发程序 |

## 九、总结

| 主题 | 核心结论 |
|------|----------|
| 排号自旋锁（Ticket Lock） | 通过原子 fetch-and-add 实现公平排队，保证有界等待 |
| 细粒度锁 | 不同资源分配独立锁，提升并发度 |
| 锁的正确性 | 依赖所有线程的协作，任何一个绕过锁的访问都会导致错误 |
| 内核自旋锁 | 必须配合中断开关（push_off/pop_off）防止中断引发的死锁 |
| 用户态自旋锁性能 | 长临界区下自旋浪费 CPU；yield 减少空转但引入大量上下文切换 |
| Futex | 两阶段锁：Fast Path 原子指令 + Slow Path 系统调用阻塞；`futex_wait` 的原子性检查消除 Lost Wakeup |
| pthread_mutex | 实际工程中首选；底层基于 Futex，兼顾性能与正确性 |
| 并发数据结构 | 从 One Big Lock 到细粒度锁、近似计数器，逐步提升可扩展性 |

> **核心认知**：软件纯软件实现互斥极难且在现代硬件下不正确；硬件原子指令 + 操作系统配合 + 精心的用户态设计，共同构成了正确、高效的互斥机制体系。
