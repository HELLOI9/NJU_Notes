# 第2章 关系数据库 — 知识点整理

## 一、背景与概述

关系数据模型由 IBM 的 **E. F. Codd**（1981年图灵奖）于 1970 年提出，论文标题为 *The Relational Model of Data for Large Shared Data Banks*。

### 名词术语对应关系

| 关系模型理论 | 关系数据库（SQL） | 文件系统 |
|---|---|---|
| 关系 relation | 表 table | 文件 |
| 属性 attribute | 列 column | 字段 field |
| 元组 tuple | 行 row | 记录 record |
| 关系模式 schema | 表头 table heading | 记录型 |
| 码 / 候选码 / 主码 | unique / primary key | — |

## 二、关系数据结构及形式化定义（2.1）

### 2.1.1 关系

#### 1. 域（Domain）

> **定义2.1**：域是一组具有**相同数据类型**的值的集合。

- 每个域有唯一的**域名**，同一域中元素互不相同。
- 示例：
  - $D_1 = \text{SUPERVISOR} = \{张清玫, 刘逸\}$
  - $D_2 = \text{SPECIALITY} = \{计算机专业, 信息专业\}$
  - $D_3 = \text{POSTGRADUATE} = \{李勇, 刘晨, 王敏\}$

---

#### 2. 笛卡尔积（Cartesian Product）

> **定义2.2**：给定一组域 $D_1, D_2, \cdots, D_n$（允许相同），其笛卡尔积为：
> $$D_1 \times D_2 \times \cdots \times D_n = \{(d_1, d_2, \cdots, d_n) \mid d_i \in D_i,\ i=1,2,\ldots,n\}$$

**相关概念：**

| 概念 | 说明 |
|---|---|
| **元组**（n元组） | 笛卡尔积中的每个元素 $(d_1, d_2, \ldots, d_n)$ |
| **分量** | 元组中的每一个值 $d_i$ |
| **基数** | 各域基数之积：$M = \prod_{i=1}^{n} m_i$ |

**表示方法：**
- **集合表示法**：列举所有 n 元组
- **二维表表示法**：每行一个元组，每列一个域，第一行为表头

---

#### 3. 关系（Relation）

> **定义2.3**：笛卡尔积 $D_1 \times D_2 \times \cdots \times D_n$ 的**子集**称为在这些域上的关系，记为 $R(D_1, D_2, \ldots, D_n)$，即 $R \subseteq D_1 \times D_2 \times \cdots \times D_n$

**目/度（Degree）：**

| n 的值 | 名称 |
|---|---|
| n = 1 | 单元关系（Unary relation） |
| n = 2 | 二元关系（Binary relation） |
| n > 2 | n 元关系 / 多元关系 |

**属性（Attribute）：**
- 关系二维表中的每一列称为一个属性，有唯一的**属性名**
- n 目关系必有 n 个属性

**码（Key）与候选码（Candidate Key）：**
- **候选码**：某属性组能唯一标识一个元组，且其任意真子集不能，则称该属性组为候选码
  - 最极端情况：所有属性构成候选码，称为**全码（All-key）**
  - 候选码是**语义范畴**的概念，不能仅凭当前元组集合判断
- **主码（Primary Key）**：从候选码中选定一个，是 RDBMS（SQL）层面的概念，一张基表**最多定义一个**主码
- **主属性**：候选码中包含的属性
- **非主属性/非码属性**：不出现在任何候选码中的属性

**关系的分类：**

| 类型 | 说明 |
|---|---|
| 基本关系（基表） | 实际存在，是实际存储数据的逻辑表示 |
| 查询表 | 查询结果对应的表，数据无需持久存储，查询请求也无需保存 |
| 视图表 | 由基本表或其他视图导出的虚表，查询请求需**持久保存**在数据库中 |

**关系的六条性质：**

| 编号 | 性质 | 说明 |
|---|---|---|
| ① | **列同质性** | 每列分量来自同一个域，类型相同 |
| ② | **列可出自同域** | 不同属性可对应同一个域，但属性名必须唯一 |
| ③ | **列的无序性** | 列的顺序可任意交换，通过属性名访问 |
| ④ | **行的唯一性** | 任意两个元组在候选码属性上的取值不同 |
| ⑤ | **行的无序性** | 行的顺序可任意交换，通过属性值定位 |
| ⑥ | **分量原子性** | 每个分量必须是不可分的原子值，不允许多值属性 |

> **注**：满足以上六条性质的二维表才能称为"关系"；现实 RDBMS 产品中基本表不一定完全满足这六条。

---

### 2.1.2 关系模式（Relation Schema）

**关系的型与值：**
- **关系模式** = 关系的"型"，静态、稳定（对关系的描述）
- **关系** = 关系模式在某一时刻的"值"（关系实例），动态、随时间变化

**形式化表示：** $R(U, D, \text{DOM}, F)$

| 符号 | 含义 |
|---|---|
| $R$ | 关系名 |
| $U$ | 属性名集合 |
| $D$ | 属性所来自的域 |
| $\text{DOM}$ | 属性向域的映象集合 |
| $F$ | 属性间数据依赖关系的集合（完整性约束条件） |

**简记形式：** $R(U)$ 或 $R(A_1, A_2, \ldots, A_n)$

---

### 2.1.3 关系数据库

- 在一个应用领域中，**所有关系的集合**构成一个关系数据库
- **关系数据库的型**：关系数据库模式（所有关系模式及其他数据库对象定义）
- **关系数据库的值**：某一时刻所有关系的集合

---

### 2.1.4 关系模型的存储结构

- 关系模型是**逻辑数据模型**，不涉及物理存储
- 不同 RDBMS 有不同的物理组织方式（一表一文件，或自行管理大文件）

---

## 三、关系操作（2.2）

### 基本操作（5种）

$$\text{选择} \quad \text{投影} \quad \text{并} \quad \text{差} \quad \text{笛卡尔积}$$

### 导出操作

交、连接（$\theta$-连接、自然连接、外连接）、除

### 关系操作的核心特点

> **操作对象是关系，操作结果也是关系**（即"一切皆集合"）

### 数据操纵功能分类

| 功能 | 实现方式 |
|---|---|
| **数据查询**（单表） | 元组选择 + 属性投影 |
| **数据查询**（多表） | 先笛卡尔积/连接合并为单关系，再查询 |
| **元组插入** | 新元组构成中间关系，用**并运算**实现 |
| **元组删除** | 先查询得到旧元组构成中间关系，用**差运算**实现 |
| **元组修改** | 先删除旧元组，再插入新元组 |

### 关系数据语言分类

```
关系数据语言
├── 关系代数语言
├── 关系演算语言
│   ├── 元组关系演算语言（谓词变元基本对象是元组变量）
│   └── 域关系演算语言（谓词变元基本对象是域/属性变量）
└── 具有关系代数和关系演算双重特点的语言（SQL）
```

## 四、关系的完整性（2.3）

> 关系在**任何时刻**都必须满足完整性约束，分为三类。

---

### 2.3.1 实体完整性（Entity Integrity）

**规则2.1：**
- 若属性 $A$ 是基本关系的**主码中的属性**，则 $A$ **不能取空值（null）**
- **空值**的含义："不知道" 或 "不存在" 或 "无意义"

**说明：**

1. 实体完整性针对**基本关系**（基表）
2. 现实世界中实体可区分，需唯一性标识
3. RDBMS 以**主码**作为唯一性标识
4. 若基表未定义主码，则允许出现重复行，不满足实体完整性

**示例：**
- 学生（**学号**，姓名，性别，专业号，年龄）→ 学号不能为空
- 选修（**学号，课程号**，成绩）→ 学号和课程号均不能为空

---

### 2.3.2 参照完整性（Referential Integrity）

#### 外码定义

> **定义2.5**：设 $F$ 是基本关系 $R$ 的一个或一组属性，但不是 $R$ 的码。若 $F$ 与基本关系 $S$ 的主码 $K_S$ 相对应，则称 $F$ 是 $R$ 的**外码（Foreign Key）**。
> - $R$：**参照关系（Referencing Relation）**
> - $S$：**被参照关系（Referenced Relation）/ 目标关系**

**注意：**
- $R$ 和 $S$ 可以是**同一个关系**（自引用）
- 外码与对应主码必须定义在**同一个域**上
- 外码名与主码名**不必相同**

#### 三类引用示例

| 示例 | 说明 | 外码 |
|---|---|---|
| 例2.1 | 学生引用专业 | 学生.专业号 → 专业.专业号 |
| 例2.2 | 选修引用学生和课程 | 选修.学号 → 学生.学号；选修.课程号 → 课程.课程号 |
| 例2.3 | 学生内部自引用 | 学生.班长学号 → 学生.学号 |

#### 参照完整性规则（规则2.2）

> 对于参照关系 $R$ 中每个元组在外码 $F$ 上的值，必须满足以下之一：
> - **取空值**（外码的每个属性均为空）
> - **等于被参照关系 $S$ 中某个元组的主码值**

**特殊情况（例2.2）：** 当外码同时是主属性时（如选修关系中的学号、课程号），**不能取空值**，只能取被参照关系中已存在的主码值。

---

### 2.3.3 用户定义的完整性（User-defined Integrity）

针对具体应用的语义约束，由 RDBMS 统一支持，常见类型：

| 约束类型 | 示例 |
|---|---|
| **唯一性约束** | 身份证号唯一 |
| **非空约束** | 姓名、性别不能为空 |
| **取值范围约束** | 性别 ∈ {男, 女}；学分 ∈ {1,2,3,4} |

---

## 五、三类完整性约束对比总结

| 类型 | 约束对象 | 是否自动支持 | 核心要求 |
|---|---|---|---|
| **实体完整性** | 主码属性 | ✅ RDBMS 自动支持 | 主码属性不为空，主码值唯一 |
| **参照完整性** | 外码属性 | ✅ RDBMS 自动支持 | 外码值为空或等于被参照关系中某主码值 |
| **用户定义完整性** | 特定属性 | ⚙️ 用户定义，系统执行 | 满足特定语义约束 |

> 实体完整性和参照完整性合称**"关系的两个不变性"**。
