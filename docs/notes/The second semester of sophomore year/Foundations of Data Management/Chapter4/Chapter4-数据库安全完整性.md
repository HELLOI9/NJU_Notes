# 第4章 数据库安全性

## 1 数据库安全性概述

### 1.1 为什么需要安全性

数据库的一大特点是**数据共享**，但数据共享必然带来安全问题，因此数据库中的数据共享不能是无条件的共享。典型的敏感数据场景：

- 军事秘密、国家机密
- 新产品实验数据
- 市场需求分析、销售计划
- 客户档案、医疗档案、银行储蓄数据

**定义：** 数据库的安全性是指**保护数据库以防止不合法使用所造成的数据泄露、更改或破坏**。系统安全保护措施是否有效是数据库系统主要的性能指标之一。

### 1.2 数据库面临的威胁来源

| 威胁类型 | 对应保护措施 |
|---|---|
| 非法用户 | 安全性保护（权限管理） |
| 非法数据 | 完整性保护 |
| 各种故障或误操作 | 故障恢复 |
| 多用户并发访问 | 并发控制 |

### 1.3 数据库不安全因素

1. **非授权用户的恶意存取和破坏**
   - 黑客猎取用户名和口令，假冒合法用户偷取、修改、破坏数据
   - 对策：用户身份鉴别、存取控制、视图

2. **重要或敏感数据被泄露**
   - 黑客盗窃机密数据
   - 对策：强制存取控制（MAC）、数据加密存储和传输、审计日志分析

3. **安全环境的脆弱性**
   - 数据库安全依赖于整个计算机系统安全（硬件、OS、网络）
   - 对策：建立可信（Trusted）计算机系统的概念和标准

## 2 安全标准

### 2.1 TCSEC（橙皮书）

- **颁布时间：** 1985 年，美国国防部（DoD）
- **全称：** DoD 可信计算机系统评估准则（Trusted Computer System Evaluation Criteria）
- **安全级别划分（由低到高）：**

| 安全级别 | 名称 | 要点 |
|---|---|---|
| D | 最小保护 | 一切不符合更高标准的系统 |
| C1 | 自主安全保护 | 用户与数据分离；DAC；保护/限制用户权限传播 |
| C2 | 受控的存取保护 | DAC 进一步细化；个人身份注册；审计；资源隔离 |
| B1 | 标记安全保护 | 对数据加标记；实施 MAC；审计 |
| B2 | 结构化保护 | 形式化安全策略模型；对所有主体/客体实施 DAC + MAC |
| B3 | 安全域 | TCB 满足访问监控器要求；更强审计跟踪；系统恢复 |
| A1 | 验证设计 | B3 级保护 + 形式化设计说明和验证 |

> 各安全级别之间具有**偏序向下兼容**的关系（高级别包含低级别的所有要求）。

### 2.2 CC 标准（通用准则）

- 背景：1993 年，CTCPEC、FC、TCSEC、ITSEC 联合行动，形成 CC（Common Criteria）项目
- 1999 年 CC V2.1 被 ISO 采纳为国际标准（ISO 15408），2001 年被中国采用
- **目前 CC 已基本取代 TCSEC，成为评估信息产品安全性的主要标准**

CC 标准结构（三部分相互依存）：

1. **简介和一般模型** — 术语、基本概念、评估框架
2. **安全功能要求** — 类、子类、组件
3. **安全保证要求** — 保证类、子类、组件；评估保证级（EAL1～EAL7）

**EAL 与 TCSEC 对照：**

| EAL 级别 | 定义 | 近似对应 TCSEC |
|---|---|---|
| EAL1 | 功能测试 | — |
| EAL2 | 结构测试 | C1 |
| EAL3 | 系统地测试和检查 | C2 |
| EAL4 | 系统地设计、测试和复查 | B1 |
| EAL5 | 半形式化设计和测试 | B2 |
| EAL6 | 半形式化验证的设计和测试 | B3 |
| EAL7 | 形式化验证的设计和测试 | A1 |

### 2.3 中国计算机安全标准

**GB 17859-1999**（参照 TCSEC）：

| GB 17859-1999 分级 | TCSEC 分级 |
|---|---|
| 第 1 级：自主安全保护级 | C1 |
| 第 2 级：系统审计保护级 | C2 |

- **GJB 2646-96**：军用计算机安全评估标准（参照 TCSEC）

## 3 数据库安全性控制

### 3.1 计算机系统安全模型（多层保护）

安全措施是**一级一级层层设置**的：

```
用户 ──→ DBMS ──→ OS ──→ DB
用户标识和鉴别 | 数据库安全保护 | 操作系统安全保护 | 数据密码存储
```

- **用户层**：系统根据用户标识鉴定身份，合法用户才能进入系统
- **DBMS 层**：进行存取控制，只允许用户执行合法操作
- **OS 层**：操作系统自身的保护措施
- **数据层**：数据以密码形式存储到数据库

### 3.2 DBMS 安全性控制模型（存取控制流程）

存取控制流程：

1. **身份鉴别**：对提出 SQL 访问请求的用户进行身份验证，防止不可信用户使用系统
2. **SQL 处理层**：依次进行**自主存取控制（DAC）** 和**强制存取控制（MAC）**，进一步可进行推理控制
3. **审计**：对用户访问行为和系统关键操作进行审计，对异常用户行为进行简单入侵检测

### 3.3 常用安全控制方法

- 用户标识和鉴别
- 存取控制（DAC + MAC）
- 视图
- 审计
- 数据加密

---

### 3.4 用户身份鉴别（Identification & Authentication）

**定义：** 系统提供的**最外层**安全保护措施。

**用户标识：** 由**用户名**和**用户标识号**组成，用户标识号在系统整个生命周期内唯一。

**鉴别方法：**

| 方法 | 说明 | 特点 |
|---|---|---|
| 静态口令鉴别 | 用户自己设定，静态不变的口令 | 简单，但安全性较低，容易被截获 |
| 动态口令鉴别 | 每次使用动态产生的新口令（一次一密，如短信验证码、令牌） | 安全性较高，防重放攻击 |
| 生物特征鉴别 | 指纹、虹膜、掌纹等生物特征认证 | 难以伪造，但成本高 |
| 智能卡鉴别 | 不可复制的硬件，内置集成电路芯片，具有硬件加密功能 | 双因素认证，安全性高 |

> **补充（PPT 未涉及）：** 现代系统常采用**多因素认证（MFA）**，组合以上两种或多种方式，如"口令 + 动态令牌"或"口令 + 指纹"，进一步提升安全性。

---

### 3.5 存取控制

**存取控制机制的两个核心组成：**

1. **定义用户权限**：将用户权限登记到数据字典中（称为"安全规则"或"授权规则"）
   - 用户对某一数据对象的操作权力称为**权限**
2. **合法权限检查**：用户发出操作请求后，DBMS 查找数据字典，根据安全规则进行检查

**两种主要存取控制方法：**

| 类型 | 英文 | 起始安全级别 | 核心思想 |
|---|---|---|---|
| 自主存取控制 | Discretionary Access Control（DAC） | C1 | 用户自主管理权限，权限可转授 |
| 强制存取控制 | Mandatory Access Control（MAC） | B1 | 系统强制为数据对象和用户打密级标记，按规则控制访问 |

---

### 3.6 自主存取控制（DAC）

**特点：**

- 用户对不同的数据对象有不同的存取权限
- 不同的用户对同一对象也有不同的权限
- 用户可将其拥有的存取权限转授给其他用户

**可控制的对象与权限：**

| 对象类型 | 对象 | 操作类型 |
|---|---|---|
| 数据库模式 | 模式 | CREATE SCHEMA |
| 数据库模式 | 基本表 | CREATE TABLE，ALTER TABLE |
| 数据库模式 | 视图 | CREATE VIEW |
| 数据库模式 | 索引 | CREATE INDEX |
| 数据 | 基本表和视图 | SELECT, INSERT, UPDATE, DELETE, REFERENCES, ALL PRIVILEGES |
| 数据 | 属性列 | SELECT, INSERT, UPDATE, REFERENCES, ALL PRIVILEGES |

**用户存取权限定义规则：**

1. 一个客体的**属主（Owner）**自动拥有该客体上的所有存取权限
2. 拥有权限的用户可自主将权限传授给其他已注册用户
3. 用户只能通过以上两种方式获得权限

**访问控制检查流程（对每条 SQL 命令）：**

1. 判断用户 S 是否是数据库管理员 → 是则通过
2. 判断 S 是否是对象 O 的拥有者 → 是则通过
3. 判断 S 是否在 O 上拥有方式 A 所需的存取权限 → 有则通过，没有则权限检查失败

**DAC 的缺点：** 可能存在数据的"**无意泄露**"，因为机制仅通过存取权限控制，而数据本身无安全性标记。

### 3.7 授权：授予（GRANT）与回收（REVOKE）

#### 3.7.1 GRANT 语句

```sql
GRANT <权限> [, <权限>] ...
ON <对象类型> <对象名> [, <对象类型> <对象名>]...
TO <用户> [, <用户>] ...
[WITH GRANT OPTION];
```

- **授权用户（发出 GRANT 的用户）：** DBA、数据库对象的属主、拥有该权限及继续授权许可的用户
- **被授权用户：** 一个或多个具体用户、角色（ROLE）、PUBLIC（全体用户）
- **WITH GRANT OPTION：** 被授权用户同时获得"继续授权"许可，可将权限再授予他人；**不允许循环授权**

**示例（与 PPT 例题对齐）：**

```sql
-- [例4.1] 把查询Student表权限授给用户U1
GRANT SELECT ON TABLE Student TO U1;

-- [例4.2] 把Student表和Course表的全部权限授予U2和U3
GRANT ALL PRIVILEGES ON TABLE Student, Course TO U2, U3;

-- [例4.3] 把SC表的查询权限授予所有用户
GRANT SELECT ON TABLE SC TO PUBLIC;

-- [例4.4] 把查询Student表和修改学生学号的权限授给U4
-- 注：对属性列的授权，必须明确指出属性列名
GRANT UPDATE(Sno), SELECT ON TABLE Student TO U4;

-- [例4.5] 把SC表的INSERT权限授予U5，并允许继续授权
GRANT INSERT ON TABLE SC TO U5 WITH GRANT OPTION;

-- [例4.6] U5将INSERT权限及传播权限授予U6
GRANT INSERT ON TABLE SC TO U6 WITH GRANT OPTION;

-- [例4.7] U6将INSERT权限授予U7，但U7不能再传播
GRANT INSERT ON TABLE SC TO U7;
```

**权限传播链（例4.5～4.7）：**
```
DBA → U5（可传播）→ U6（可传播）→ U7（不可传播）
```

#### 3.7.2 REVOKE 语句

```sql
REVOKE <权限> [, <权限>] ...
ON <对象名> [, <对象名>]...
FROM <用户> [, <用户>] ...
[CASCADE | RESTRICT];
```

- **CASCADE**：连锁回收。若用户 A 已将权限转授给 B，则同时从 A 和 B 处回收
- **RESTRICT**：仅当不存在连锁回收问题时才执行，否则拒绝执行

**示例：**

```sql
-- [例4.8] 收回U4修改学生学号的权限
REVOKE UPDATE(Sno) ON Student FROM U4;

-- [例4.9] 收回所有用户对SC表的查询权限
REVOKE SELECT ON SC FROM PUBLIC;

-- [例4.10] 收回U5对SC表的INSERT权限（CASCADE连锁回收U6、U7）
REVOKE INSERT ON SC FROM U5 CASCADE;
```

> **注意：** 若 U6/U7 还从其他用户获得了 SC 表的 INSERT 权限，系统只回收直接或间接从 U5 处获得的权限，他们仍可能保留该权限。

#### 3.7.3 数据库模式权限

```sql
CREATE USER <username> [WITH] [DBA | RESOURCE | CONNECT];
```

| 权限 | 可创建用户 | 可创建模式 | 可创建表/视图 | 可登录数据库 |
|---|---|---|---|---|
| DBA | ✓ | ✓ | ✓ | ✓ |
| RESOURCE | ✗ | ✗ | ✓ | ✓ |
| CONNECT | ✗ | ✗ | ✗ | ✓（需有相应权限） |

- **CONNECT（默认）：** 只能登录，不能创建用户、模式、基本表
- **RESOURCE：** 能创建基本表和视图，成为所创建对象的属主；不能创建模式和新用户
- **DBA：** 超级用户，拥有一切权限

### 3.8 数据库角色

**定义：** 角色是**被命名的一组与数据库操作相关的权限**（权限的集合）。用于为一组具有相同权限的用户简化授权过程。

#### 角色相关 SQL 语法

```sql
-- 创建角色
CREATE ROLE <角色名>;

-- 给角色授权
GRANT <权限> [, <权限>]... ON <对象类型> <对象名> TO <角色> [, <角色>]...;

-- 回收角色权限
REVOKE <权限> [, <权限>]... ON <对象类型> <对象名> FROM <角色> [, <角色>]...;

-- 将角色授予用户或其他角色（WITH ADMIN OPTION：继续授权许可）
GRANT <角色> [, <角色>]... TO <角色_or_用户> [, <角色_or_用户>]... [WITH ADMIN OPTION];

-- 从用户或角色回收角色
REVOKE <角色> [, <角色>]... FROM <角色_or_用户> [, <角色_or_用户>]...;
```

**WITH GRANT OPTION 与 WITH ADMIN OPTION 的区别：**

| 选项 | 适用范围 | 说明 |
|---|---|---|
| WITH GRANT OPTION | 对象权限（如 SELECT/INSERT 等） | 允许被授权者将对象权限转授他人 |
| WITH ADMIN OPTION | 系统权限 / 角色授予 | 允许被授权者将角色或系统权限再授予他人 |

**示例（例4.11）：**

```sql
-- Step1: 创建角色R1
CREATE ROLE R1;

-- Step2: 授予R1权限
GRANT SELECT, UPDATE, INSERT ON TABLE Student TO R1;

-- Step3: 将角色授予用户
GRANT R1 TO 王平, 张明, 赵玲;

-- 批量回收：一次性回收王平的所有R1权限
REVOKE R1 FROM 王平;

-- 角色权限修改（例4.12）：增加DELETE权限
GRANT DELETE ON TABLE Student TO R1;
-- 张明和赵玲自动获得DELETE权限

-- 角色权限修改（例4.13）：减少SELECT权限
REVOKE SELECT ON TABLE Student FROM R1;
-- 张明和赵玲的SELECT权限自动被回收
```

**一个角色的权限 = 直接授予该角色的权限 + 通过其他角色授予该角色的权限**

### 3.9 强制存取控制（MAC）

**定义：** 强制存取控制（Mandatory Access Control，MAC），从 B1 级开始要求，提供更高程度的安全性。对数据本身进行密级标记，标记与数据不可分离。

**适用场景：** 对数据有严格而固定密级分类的部门，如军事部门、政府部门。

#### 主体与客体

- **主体（Subject）**：系统中的活动实体，包括实际用户和代表用户的各进程
- **客体（Object）**：系统中的被动实体，受主体操纵，包括文件、基本表、索引、视图、存储过程等

#### 敏感度标记（Label）

DBMS 为每个主体和客体实例指派一个敏感度标记，分为：

| 标记 | 名称 |
|---|---|
| TS | 绝密（Top Secret） |
| S | 机密（Secret） |
| C | 可信（Confidential） |
| P | 公开（Public） |

支配次序：**TS ≥ S ≥ C ≥ P**

- **主体的敏感度标记**称为**许可证级别（Clearance Level）**
- **客体的敏感度标记**称为**密级（Classification Level）**

#### 强制存取控制规则（Bell-LaPadula 模型核心）

| 规则 | 名称 | 条件 | 说明 |
|---|---|---|---|
| ① | **下读（Read Down）** | 主体许可证级别 ≥ 客体密级 | 主体才能**读取**客体 |
| ② | **上写（Write Up）** | 主体许可证级别 ≤ 客体密级 | 主体才能**写入**客体 |

> **设计意图：** 下读规则防止高密级用户将高密级数据读取后泄露；上写规则防止低密级用户将低密级数据写入高密级对象（防止"特洛伊木马"攻击）。

#### DAC + MAC 组合使用

- 实现 MAC 时**必须首先实现 DAC**（较高级别包含较低级别的所有保护）
- 安全检查顺序：**SQL 语法分析 & 语义检查 → DAC 检查 → MAC 检查 → 继续**
- 只有同时通过 DAC 和 MAC 检查的数据对象才可被存取

## 4 视图机制

**作用：** 把需要保密的数据对无权存取的用户**隐藏**起来，对数据提供一定程度的安全保护；间接实现支持存取谓词的用户权限定义。

**示例（例4.14）：**

```sql
-- Step1: 建立计算机系学生的视图
CREATE VIEW CS_Student AS
SELECT * FROM Student WHERE Sdept = 'CS';

-- Step2: 在视图上授权
GRANT SELECT ON CS_Student TO 王平;
GRANT ALL PRIVILEGES ON CS_Student TO 张明;
```

**视图机制的安全性优缺点：**

| 优点 | 缺点 |
|---|---|
| 将行级/列级权限控制与 SQL 授权机制结合 | 只能提供数据独立性保护，无法防止推理攻击 |
| 可限制用户只访问特定行/列数据 | 视图本身不能替代强制存取控制 |
| 实现简单，不改变数据存储结构 | |

> **补充（PPT 未涉及）：** 视图可实现**行级安全性**（如上例限制只看 CS 系学生）和**列级安全性**（只暴露特定列）。在 PostgreSQL 等系统中还有**行级安全策略（Row Level Security，RLS）**，可在表级别对每一行自动应用访问策略，无需视图。

## 5 审计（Audit）

**定义：** 启用专用的**审计日志（Audit Log）**，将用户对数据库的所有操作记录在上面；审计员利用审计日志监控数据库中的各种行为，找出非法存取数据的人、时间和内容。

**要求：** C2 以上安全级别的 DBMS 必须具有审计功能。

**可选性：** 审计很费时间和空间，DBA 可灵活打开或关闭审计功能，主要用于安全性要求较高的部门。

### 5.1 审计事件类型

| 类型 | 说明 |
|---|---|
| 服务器事件 | 数据库服务器的启停、配置文件重新加载等 |
| 系统权限 | 对系统结构或模式对象进行操作的审计（通过系统权限获得的操作） |
| 语句事件 | 对 SQL 语句的审计，包括 DDL、DML、DQL、DCL |
| 模式对象事件 | 特定模式对象（表、视图、存储过程/函数）上的 SELECT 或 DML 操作 |

### 5.2 审计级别

| 级别 | 设置人员 | 监测内容 |
|---|---|---|
| 用户级审计 | 任何用户 | 用户针对自己创建的表和视图进行审计 |
| 系统级审计 | 仅 DBA | 监测登录成功/失败、授权/收回操作、数据库级权限下的操作 |

### 5.3 审计功能

- 提供多种审计查阅方式
- 多套审计规则（一般在数据库初始化时设定）
- 审计分析和报表功能
- 审计日志管理：

   - 审计日志必须先转储后删除
   - 对转储文件提供完整性和保密性保护
   - 只允许审计员查阅和转储，不允许任何用户新增/修改审计记录

- 提供查询审计设置及审计记录信息的专门视图

### 5.4 审计语句

```sql
-- [例4.15] 对修改SC表结构或数据的操作进行审计
AUDIT ALTER, UPDATE ON SC;

-- [例4.16] 取消对SC表上ALTER和UPDATE操作的审计
NOAUDIT ALTER, UPDATE ON SC;
```

> **补充（PPT 未涉及）：** 审计日志本身也需要保护，防止攻击者在入侵后篡改日志。现代数据库（如 Oracle、SQL Server）支持将审计日志写入独立的、受保护的存储区，甚至使用区块链技术保证日志不可篡改。

## 6 数据加密

**作用：** 防止数据库中数据在**存储**和**传输**中失密的有效手段。

**基本思想：** 根据一定算法将原始数据（**明文，Plain text**）变换为不可直接识别的格式（**密文，Cipher text**）。

### 6.1 存储加密

| 类型 | 说明 | 特点 |
|---|---|---|
| 透明存储加密 | 内核级加密，写磁盘时加密、授权读取时解密，对用户完全透明；只需在建表时声明需加密字段 | 性能较好，安全完备性高，应用无需修改 |
| 非透明存储加密 | 通过多个加密函数在应用层实现 | 灵活但需修改应用程序 |

### 6.2 传输加密

| 类型 | 加密层次 | 说明 | 缺点 |
|---|---|---|---|
| 链路加密 | 链路层 | 报文和报头均加密，每个链路节点解密再加密 | 节点处数据为明文，节点安全性要求高 |
| 端到端加密 | 应用层（发送端加密、接收端解密） | 只加密报文，不加密报头 | 所需密码设备少，但容易被非法监听者通过报头获取敏感信息 |

> **补充（PPT 未涉及）：**

> - **TLS/SSL**：目前主流数据库（MySQL、PostgreSQL、Oracle 等）均支持使用 TLS/SSL 协议进行端到端加密传输，是工程实践中最常用的传输加密方式。
> - **加密算法分类：** 对称加密（如 AES-256，速度快，适合大量数据存储加密）和非对称加密（如 RSA，适合密钥交换）通常结合使用。
> - **密钥管理**：加密系统的安全性不仅取决于算法，更取决于密钥管理（密钥生成、存储、分发、更新、撤销）。密钥泄露相当于加密失效。

## 7 其他安全性保护

### 7.1 推理控制（Inference Control）

**定义：** 避免用户利用能够访问的数据，通过推理**得知更高密级的数据**。

**场景举例：** 用户无权访问某字段，但可通过多次合法聚合查询（如求均值、方差）推断出该字段的具体值。

**对策：**

- **查询审计**：检测可疑的多次相关查询
- **数据扰动**：在查询结果中加入随机噪声（如差分隐私技术）
- **查询限制**：限制可以执行的查询组合，或限制查询结果集的最小规模

### 7.2 隐蔽信道（Covert Channel）

**定义：** 利用系统中不被安全策略覆盖的**间接信息传递路径**来传递敏感信息，绕过安全机制。

**类型：**

- **存储隐蔽信道**：通过修改共享存储资源（如文件大小、磁盘占用）传递信息
- **时序隐蔽信道**：通过响应时间差异传递信息

### 7.3 数据隐私保护（Data Privacy）

**定义：** 描述个人控制其不愿他人知道或他人不便知道的个人数据的能力。

**范围：** 数据**收集 → 存储 → 处理 → 发布**各阶段。

> **补充（PPT 未涉及）：** 数据隐私保护的重要技术：

> - **数据脱敏（Data Masking）**：将敏感数据（如身份证号、手机号）进行处理（如掩码、随机化）后再使用，用于非生产环境或对外数据共享
> - **差分隐私（Differential Privacy）**：通过向查询结果中加入数学意义上可控的随机噪声，使得统计查询无法精确还原个体信息，同时保证统计分析结果的可用性
> - **数据匿名化（k-匿名、l-多样性等）**：对数据集进行处理，使得每条记录至少与另外 k-1 条记录不可区分
> - **隐私计算**：联邦学习、安全多方计算（MPC）、同态加密等技术允许在不暴露原始数据的前提下进行数据计算

## 8 小结与知识点对照

### 数据库安全性技术总览

| 技术 | 核心目标 | 主要手段 |
|---|---|---|
| 用户身份鉴别 | 确认用户合法性 | 静态/动态口令、生物特征、智能卡 |
| 自主存取控制（DAC） | 用户自主管理访问权限 | GRANT / REVOKE / ROLE |
| 强制存取控制（MAC） | 系统强制控制，防无意泄露 | 密级标记、下读/上写规则 |
| 视图机制 | 隐藏敏感数据 | CREATE VIEW + GRANT |
| 审计 | 事后追责与监控 | AUDIT / NOAUDIT、审计日志 |
| 数据加密 | 防止数据被截获后泄密 | 存储加密（透明/非透明）、传输加密（链路/端到端） |
| 推理控制 | 防止间接推导高密级数据 | 查询限制、数据扰动 |
| 数据隐私保护 | 保护个人数据全生命周期 | 脱敏、差分隐私、匿名化 |

### 关键概念辨析

| 概念对 | 区别 |
|---|---|
| DAC vs MAC | DAC 由用户自主控制权限（可转授）；MAC 由系统强制，基于密级标记，用户不可自主更改 |
| 主体 vs 客体 | 主体是操作发起方（用户/进程）；客体是被操作方（表/视图/文件等） |
| 许可证级别 vs 密级 | 许可证级别是**主体**的标记；密级是**客体**的标记 |
| 下读 vs 上写 | 下读：主体许可 ≥ 客体密级才能读；上写：主体许可 ≤ 客体密级才能写 |
| WITH GRANT OPTION vs WITH ADMIN OPTION | GRANT OPTION 是对象权限的转授权；ADMIN OPTION 是系统权限/角色的转授权 |
| CASCADE vs RESTRICT（REVOKE） | CASCADE 连锁回收下级授权；RESTRICT 存在连锁回收问题时拒绝执行 |
| 链路加密 vs 端到端加密 | 链路加密报头+报文都加密但节点暴露；端到端只加密报文，路由信息可见 |
| 审计 vs 存取控制 | 存取控制是预防性的（事前）；审计是追溯性的（事后） |
